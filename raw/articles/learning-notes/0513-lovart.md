# 0513 - 【学习】研究下 Lovart

**核心感受**
- 技术上 lovart 没有核心技术，开发和实践上可能有一些最佳实践的积累，但是也不是技术问题，只是人力和效率问题。
- 产品上，对设计场景如何配合 AGENT，我觉得 lovart 是有很多想法的
  - 首先**是要交付产物，不是交付 clip**，那么就需要一个通用容器去承载一个设计项目/工程。这个容器是有最佳实践的、即 figma
  - 其次把 AGENT 当做实习生去设计产品，设计产物（clip），用户只需要关注理念（丰富的知识库），理念对齐后，不需要关注 clip 怎么被生成的（用户对 Prompt  和工具执行无感），要能够方便的地指导 AGENT 去调整（Clip 的方面的语义上的后编辑）。
- 最后是非常好的工程能力，交付了很好很完整的产品，很棒，很完整，牛批

**更多判断 / 有待回答的问题**
- Lovart 对我的影响是什么？ - 应该要更高效地把 AGENT 的能力扩展出去！
  - AGENT 产品任务上，大家的思路是趋同的，效率提升上大概率有一些最佳实践我们没有找到
  - AGENT 的能力丰富，比单个模态下的工具完整度可能要重要很多
- 未来多模态交互的形态是什么？ 双流合一和画布？ 怎么融合？  - 未知，等明天再和大家聊聊

# 感受和想法（看这part即可，时间多可以看下我附上的 case）
## 技术上的想法
- 从 AGENT 的角度，越体验越失落，一方面是感觉 lovart 的技术架构和主要的 AGENT 实现并没有什么新奇的，没有收获到什么新的认知（甚至不如豆包超能创意给我的启发大），如果想了解 lovart 的技术实现可以直接看以下的文档：
  - 即梦Agent-技术规划
  - ChatGPT Canvas对话请求分析
  - Agent流式协议设计
- 另一方面我也在思考，AGENT 技术的优化到底是什么，从架构来看，AGENT 有三个部分，host、tools、resources，首先 HOST 的优化，与 AGENT 没关系，做不了。tools 的迭代，和 AGENT 整体架构关系不大，只是要维护一套 tool 的注册和调度机制，弄好了维护即可。目前结合近期的思考，可能会有两个方向：
  - 更多知识（Resource ）的积累，这里的 Resource 应该有几部分：
    - **工具的最佳实践和良好用法**
    - 用户在**特殊场景需要的设计知识或参考**，在 lovart 里体现为蓝色的工具（搜索、知识库、灵感、数据库、风格推荐等），这部分内容可能是可以通过某些特殊的 online learning 的工程方法，拿到并固化下来的。并且也许可能构建出一些独特壁垒

## 产品设计上
### AGENT 的产品设计上
- AGENT 工具定义上，太丰富了，这种丰富感让我觉得自己是小丑，不仅有你能想到的图、视频、知识对齐等工具，还有 3D 生成，音乐生成，声音合成等等，这里有两个点，我觉得值得 highlight 一下
  - **不考虑兼容性，先考虑能力完整性**，举例：
    - 音乐生成&配音的结果是直接配在视频里的，没有单独的容器，3D 生成是单独的弹窗展示产物，视频的结果也是点击之后弹窗看视频（在画布中的表现）
    - LLM 产物 & 知识性的工具结果展示是很通用的，会导致某些工具信息展示不全，但是可能他们不在乎。
  - **MCP server 的注册和开发一定是被他们研究出了一个高效的方法**，这里我和刑政再沟通一下；
- 在 AGENT流 的产品定义上，我觉得整体上，没有什么事情是让人眼前一亮的，MVP工具的模块化设计是共识的（不信的话请翻阅我和晓晖的二期需求）。在某些信息模块获取的更好，可能也只是相关工具使用的 O3 或者 4o 进行的理解和生成，并没有什么特别出彩的地方。
  - 有一点不好的，每次和AGENT 的交互，稳定触发漫长冗长的思考，太磨叽了，感觉 PE 还是写的有点小问题
- 在画布和 AGENT 流的联动上，我觉得有很多细节做的很有想法：
  - AGENT 每一张图自动被命名了，这个其实很适合做上下文指代，指代图片目前需要在 Prompt 里拼一个对应图片的 URI，其实少了当时推理的信息，这个标题类似于一种 summary，可以为后面降低一些推理难度

  - 丝滑的引用！ 抄了！
    - 在测试中 我几乎不会使用 edit 都是使用 add to chat 来完成任务，尽管edit理论上会更快，但是我会觉得表达上更有负担，我愿意等待一会

### 画布的产品设计上
太屌了，太完整了，好多有趣的事情
- **好玩的图层。**这里的图层功能实际上解决了很多问题，产物的生产顺序表征，管理，等等，实际上这里的图层可以看作是一个生成流的缩略

- 资产、编辑和画布的联动协同
  - 资产到编辑非常丝滑，直接可以在资产的最上层完成第一步的编辑操作
  - 编辑之后，可以在内层的画布，完成编辑操作的流动，我和羿铭都觉得太帅了
  - 另外编辑操作似乎可以在内层页面进行固化，还可以进行后续的 RPA 流程？

- 其他完成度高的细节（点击 insert image 之后的图像生成器、贴心的字体、一件分图层等）
