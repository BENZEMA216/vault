# 0710
## AGENT 的第一性原理是什么？
AGENT 的第一性原理是什么呢？ - 半夜想到这个问题，感觉大概有个答案，所以翻身起来写下文档
- AGENT 的核心目标是通过 V/LLM在整体任务中提高智力总量，这句话等同于，**在任务中，LLM 实现的有效推理总 token 数。**
- 如果从效能的角度出发，那就是基于**单元任务，LLM消耗的有效 Token 数。**
让我画一张图来自我表达一下
<whiteboard token="SA0VwdQ25hWEh8bES2Yczr75nTd" width="639" height="313"/>

我们需要在任务的核心部分，尽可能地消耗最大量的 token 数，任务的核心往往是最需要智力或者理解的部分，我们应该期待 AGENT 在这个部分尽可能地推理，完成后再进行其他的部分。
我为什么这么拆分呢？ 因为在实际的 AGENT 任务中，会有很多无效的推理 token 和边缘任务，比如标点符号、一些很不需要智能但是很重要的事情，比如商业化、安审等。
当然，我知道实际上token 只要推理了就是有效的，特别是对于较难任务而言，但是我仍然希望可以通过一个概念上的有效和无效来说明我的想法。

对于创作 AGENT 来说，第一性原理是什么呢？ 有 N 个 token 是为了最终合计 M字的Prompt 在推理，那么这个 N/M的值越高，对于我们的任务而言可能就越成功。
当然这样好像仍然是不好理解的，那么我们可以再具象一下
<whiteboard token="O8eQwoXzOhWWJ9bC17fcF6pznFd" width="639" height="313"/>

- 我们把对 Prompt 的要求、对边缘逻辑的需求叠加用户的任务，经过 tokenizer 后一起输入给 VLM
- VLM 推理后，有部分 tokens 转换成任务需要的好的 Prompt，另外的部分用于维护边缘逻辑。
- Input/ Output Token 对于 LLM 来说是 顺序权重 * 数量的，但是实际推理的内容，对于整个任务来说，语义上是很不平等的，部分内容就是对整个产品来说更重要/更难的。
我希望到这儿，大家可以理解我的想法，我的想法是
<callout emoji="man-raising-hand" background-color="light-orange" border-color="light-orange">
AGENT 的核心工作是，通过任何手段，尽可能放大 VLM 在真正重要事情上的推理效能
为什么强调效能？ 因为实际产品就是对成本有要求的，不能无限消耗 tokens
</callout>

在不考虑提高底模智能水平的情况，我们有什么手段呢？
1. 上下文治理 - 现在创作 AGENT 的上下文就是一坨屎，引入了太多不具有语义信息和用途的 INPUT 了
1. Multi-Agent - 把重要但是不需要智力的任务分包出去，比如商业化和安审、或者把 Prompt 中的重复部分包出去
1. 增加 AGENT 在任务上的 token 推理数，打开/引导thinking、React、或者 deep research 的方式
1. 增加用户输入的明确程度
Context Engineering 的价值，用这样的方式可以很直白地理解
## 后训练重要吗？
- 后训练不带来新的能力，后训练不能提高单位 Token 的价值，但是可能会提高对核心任务的 Token 消耗。
- 后训练可以带来稳定性和效率提升
- 如果后训练要在更小尺寸的模型上训练，那就是自找苦吃，纯为了发论文/混口饭吃。
## 反思
- 没有想清楚 multi-agent 的用途，我们把 mutli agent 当成提高效果的手段，让子 agent 也做关键任务部分，其实是错的，应该让子 agent 作为简单任务的外包
- 没有很认真的思考上下文工作，最开始的核心都在关注单轮任务效果，其实是不对的
- 没有从最开始就关注推理的效率问题。
- AGENT 可能从最开始也应该试着争取一下单开产品或者别的方式

## 为什么AGENT需求这么痛苦？
<callout emoji="man-raising-hand" background-color="light-orange" border-color="light-orange">
0710：如果一件事情做得很痛苦，那一定是问题错了。
成熟产品的看似重要的逻辑都是影响 AGENT 推理效能的原因，应该最开始就想清楚。
</callout>

从我的痛苦感知来说，做AGNET需求比我打篮球骨折还痛苦。（这句话写于五一节后上线前）
### 反思一：AGENT不适合有耦合的业务逻辑，业务逻辑带来超多的复杂度
<quote-container>
为什么别人一个mcp server只需要做两小时，我们要做三四天？
因为耦合了业务逻辑，要支持产物分发，支持后编辑能力。
为啥别人串行并行一会儿整完了，我们要盘逻辑从白天盘到黑。
因为我们要兼容已有产物容器，并且数据互通，资产一份。
</quote-container>

- 在需求规划时，PM（特别是新来的，不了解那么多历史数据逻辑的PM，例如我和春），很难规划到复杂耦合逻辑带来的额外工作量
  - 在需求设计时，我们会倾向于，先在某一个主要的容器里做，期待这个容器里先验证，不影响其他模块的用户体验。
  - 但是为了体验的最小闭环，和某些整个产品的主张原则，我们又不得不和其他模块发生耦合。有一种既定的悲剧的美。
- 在详评时，在即梦，会出现因为范围变化导致拉不上来人的情况。比如，让AGENT支持并行、串行调用，涉及到图方向后端的人力，我直接一个晕掉，后端直接就是没有人。
- 在执行时，会出现已有逻辑对功能迭代的拖延，比如在接入视频工具的时候，AGENT后端无法顺畅地直接接入我们的视频能力，原因是生视频接口叠了太多业务逻辑，商业化、安审等。
AGENT适合单独在一个主体做，至少在验证阶段，应该要能够单独在一个独立端进行，在主端人要做冒烟咯～
### 反思二：AGENT的版本规划，不能做MVP + 敏捷迭代
AGENT 不能敏捷迭代，或者换言之，AGENT就应该是一个大版本完整产出。
- AGENT能够响应的场景数量和范围，会随着原子能力的不断接入，指数性的提升。反过来，如果AGENT少一个能力，就会少很多可响应的场景，得不偿失，AGENT应该尽可能地完成工具接入。
- AGENT的数据结构、调用方式，后端技术方案和SDK应该是能够经得起长期验证的，不然会出现：
  - 重复工作量
  - 接入某些工具倒推会要求改后端，那就老痛苦了
<quote-container>
AI 在6个月之后，会怎么样，这谁知道呢～
</quote-container>

### 想法三：选择后训练方向是一个排除法
算一个认知吧，AGENT可以做后训练，但是你要想清楚这个后训练的结果模型，要怎么样能够影响产品的直接结果指标，并且你能够获取在长时间内都生效的数据，举例：
- 训练一个Thinking模型，模型可以更准确地调用工具 - 难度大，并且你并不好枚举在长期，AGENT的任务环境会怎么样
- 训练一个图片评估模型，模型可以根据图像结果，判断要不要进行新的工具调用 - 这也是不好的，你无法提前判断成本增加和采纳提高之间的trade-off是怎么样的
-  训练一个创意模型（别管）

### 功能定义思考

上面是MCP结构下的通用定义方式，有几个假设原则，我记录下来：
- 原子能力的定义目前情况下大于其他，原子能力来源于你能搞到的、你觉得有必要的东西
- 容器定义，一般来说容器定义你可以直接选定主要场景的那个用户最多的产品的容器；然后你就把自己的产物往里套

- 用户等待时间的体验优化
- 同步任务和异步任务的设计区别
