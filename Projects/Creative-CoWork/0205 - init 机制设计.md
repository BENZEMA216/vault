# /init 机制设计方案

## 一、核心概念

### 问题定义

通用 Agent 面对太多 Skills 时行为会发散。Progressive Disclosure 解决了 token 占用，但没解决**注意力分配**——30 个 Skill 的 Level 1 元数据同时存在时，AGENT 在选择工具时会犹豫，导致不稳定。

### /init 是什么

**Skill Focus Profile** —— AGENT 层面的注意力配置，独立于 Studio（工作空间模板）。

- 不是一次性开关，而是**随场景固化而渐进演化**的机制
- 将 Skills 分为两个层级：**Active（主动）** 和 **Passive（被动）**
- AGENT 引导用户完成初始化，后续随使用持续调整

### 与现有概念的关系

```
Studio（工作空间模板）
  ├── UI 布局
  ├── 默认配置
  └── 推荐的 Skill Focus Profile ← /init 可以独立存在，也可以被 Studio 预设

/init（Skill Focus Profile）
  ├── Active Skills → Level 1 元数据始终在系统 prompt
  ├── Passive Skills → 不在系统 prompt，通过发现机制按需获取
  └── 演化规则 → 基于使用模式自动/手动调整
```

## 二、两层 Skill 模型

### Active Skills（主动层）

- Level 1 元数据（name + description）始终在系统 prompt 中
- AGENT 知道它们存在，会主动考虑使用
- 数量建议：5-10 个（控制注意力分散）

### Passive Skills（被动层）

- **不在系统 prompt 中**，AGENT 默认不知道它们存在
- 通过「发现机制」按需找到并激活
- 一旦在当前对话中被发现并使用，可临时提升为 Active

### 发现机制设计

Passive Skills 如何被找到？两种路径：

**路径 A：AGENT 主动探索**
```
AGENT 发现当前 Active Skills 无法满足用户需求
  → 调用 discover_skills() 工具
  → 返回所有 Passive Skills 的 Level 1 元数据
  → AGENT 判断哪个合适，加载其 Level 2 内容
  → 执行
```

**路径 B：用户显式请求**
```
用户："帮我加一个 Kling"  或  "有没有图片超分的能力？"
  → AGENT 调用 discover_skills(query="图片超分")
  → 返回匹配的 Skill
  → 提示用户是否要加入 Active 列表
```

### 技术实现：Prompt 组装变化

```python
def assemble_prompt(agent_type, mode, focus_profile, context_files, reminders=[]):
    prompt = ""

    # 1-3. 基础 prompt、Agent 角色、模式（不变）
    prompt += read(".prompt/system/base.md")
    prompt += read(".prompt/system/safety.md")
    prompt += read(f".prompt/agents/{agent_type}.md")
    prompt += read(f".prompt/modes/{mode}.md")

    # 4. Skill Focus Profile（新增）
    prompt += render_focus_context(focus_profile)

    # 5. Active Skills 的工具描述
    for skill in focus_profile.active_skills:
        prompt += generate_tool_description(skill)

    # 6. 发现工具（始终可用）
    prompt += generate_tool_description("discover_skills")

    # 7. 项目上下文、提醒（不变）
    for file in context_files:
        prompt += f"<context_file name='{file.name}'>\n{file.content}\n</context_file>"
    for reminder in reminders:
        prompt += read(f".prompt/reminders/{reminder}.md")

    return prompt


def render_focus_context(profile):
    """生成当前聚焦状态的描述，注入系统 prompt"""
    return f"""
## 当前场景聚焦

场景：{profile.scenario_description}
核心能力：{', '.join(profile.active_skill_names)}

你已聚焦于「{profile.scenario_description}」场景。优先使用上述核心能力完成任务。
如果当前能力无法满足需求，可以使用 discover_skills 工具探索更多能力。
"""
```

## 三、AGENT 引导式初始化流程

### 触发时机

1. **新项目首次对话** → AGENT 主动询问场景
2. **用户主动触发** → `/init` 命令
3. **AGENT 感知到场景变化** → 建议重新 init

### 引导对话示例

```
用户：我要做一个 15 秒的产品广告视频

AGENT：好的，我来帮你配置工作环境。

基于「产品广告视频」场景，我建议激活以下核心能力：
- 📝 脚本/分镜生成
- 🎬 视频生成（Sora/Minimax/Runway）
- 🖼️ 图片生成（广告素材）
- 🎵 音频/配音
- 🎞️ FFmpeg（视频处理）

以下能力将转为按需模式（需要时我会自动找到它们）：
- 代码执行、文档搜索、数据分析...

这样配置可以吗？你还有什么特别需要的能力？

用户：可以，再加一个字幕生成

AGENT：已更新。当前聚焦配置：
✅ 脚本/分镜、视频生成、图片生成、音频/配音、FFmpeg、字幕生成
💤 其他能力按需发现

配置已保存，随时可以用 /init 重新调整。
```

## 四、演化机制

### 自动演化

AGENT 在使用过程中观察模式，提出调整建议：

```
场景 1：用户反复手动触发某个 Passive Skill
  → AGENT："我注意到你经常用到图片超分，要不要把它加到核心能力里？"

场景 2：某个 Active Skill 长期未使用
  → AGENT："音频/配音能力你还没用过，要保留在核心列表里吗？"

场景 3：用户任务模式发生变化
  → AGENT："看起来你现在在做后期剪辑相关的工作，要不要切换一下焦点？"
```

### 手动演化

- `/init` → 重新初始化，完全重配
- `/init add <skill>` → 把某个 Skill 加入 Active
- `/init remove <skill>` → 把某个 Skill 移到 Passive
- `/init list` → 查看当前配置
- `/init reset` → 恢复默认（所有 Skills 平等）


### 跨会话恢复

新会话启动时：
1. 读取 `focus.yaml`
2. 恢复 Active/Passive 分层
3. 在系统 prompt 中注入场景上下文

## 六、与 Progressive Disclosure 的整合

```
                    /init 之前                    /init 之后
                ┌─────────────────┐          ┌─────────────────┐
Skill A (相关)  │ Level 1 (元数据) │   →→→    │ Level 1 (Active) │  ← 始终在 prompt
Skill B (相关)  │ Level 1 (元数据) │   →→→    │ Level 1 (Active) │  ← 始终在 prompt
Skill C (不相关)│ Level 1 (元数据) │   →→→    │ Level 0 (Passive)│  ← 不在 prompt，可发现
Skill D (不相关)│ Level 1 (元数据) │   →→→    │ Level 0 (Passive)│  ← 不在 prompt，可发现
                └─────────────────┘          └─────────────────┘

原来的 3 层变成 4 层：
Level 0: 不在 prompt（Passive，通过 discover_skills 找到）  ← 新增
Level 1: 元数据在 prompt（Active，AGENT 知道有这个能力）
Level 2: 完整指令加载（激活时）
Level 3: 资源按需加载（执行时）
```

## 七、设计决策点与方案对比

### 7.1 默认 Active 集策略

新项目首次打开、用户还没说话时，默认 Active 哪些 Skills？

| 方案 | 描述 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **A. 全部 Active** | 所有 Skills 的 Level 1 都在系统 prompt | 最大灵活性；向后兼容；不需要 init 也能工作 | 注意力分散问题仍存在；token 占用高 | 探索期、Skills 数量少时 |
| **B. 最小核心集** | 只有 read/write/bash 等基础能力 Active | 强制用户 init；行为最稳定 | 首次体验差；用户可能不知道有什么能力 | 生产环境、追求稳定性 |
| **C. 基于项目推断** | 根据项目文件类型自动推断 | 智能；无需用户操作 | 实现复杂；可能推断错误 | 有明确项目结构时 |
| **D. 分层默认** | 预设 10 个「通用能力」Active，其他 Passive | 平衡灵活性和稳定性 | 需要维护「通用能力」列表 | 推荐的折中方案 |

**方案 D 详细设计**：

```yaml
# 默认 Active 的「通用能力」
default_active:
  - read_file      # 基础
  - write_file     # 基础
  - list_files     # 基础
  - search_files   # 基础
  - bash           # 基础
  - generate_image # 创作核心
  - generate_video # 创作核心
  - ffmpeg         # 处理核心
  - web_fetch      # 信息获取
  - storyboard     # 创作核心

# 其他能力默认 Passive
# 用户 /init 后可以调整
```

### 7.2 生效粒度设计

Focus Profile 的生效范围。

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. 仅项目级** | 一个项目一个 focus.yaml，所有会话共享 | 简单；适合单一场景项目 | 同一项目内无法切换场景 |
| **B. 仅会话级** | 每次会话独立 Focus，不持久化 | 灵活；无状态 | 每次都要重新 init |
| **C. 双层设计（推荐）** | 项目级默认 + 会话级覆盖 | 最灵活；满足不同需求 | 实现稍复杂 |

**方案 C 双层设计详解**：

```
┌─────────────────────────────────────────────────────────────┐
│  项目级 Focus（持久化）                                       │
│  存储：project/.cowork/focus.yaml                           │
│  作用：所有会话的默认配置                                     │
│  修改：/init save                                           │
└─────────────────────────────────────────────────────────────┘
                              │
                         会话启动时继承
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  会话级 Focus（临时）                                         │
│  存储：内存中                                                 │
│  作用：当前会话的覆盖配置                                     │
│  修改：/init（不带 save 时只影响当前会话）                    │
└─────────────────────────────────────────────────────────────┘
```

**命令设计**：
- `/init` → 引导初始化，修改当前会话的 Focus
- `/init save` → 引导初始化，并保存到项目级
- `/init load` → 从项目级重新加载，丢弃会话级修改
- `/init reset` → 恢复默认（全部 Active 或分层默认）

### 7.3 发现机制设计

Passive Skills 如何被 AGENT 找到？

| 方案 | 实现方式 | 优点 | 缺点 | 推荐度 |
|------|---------|------|------|--------|
| **A. 返回完整列表** | `discover_skills()` → 所有 Passive Skills 的 name+desc | 简单；AGENT 自己判断 | Skills 多时列表长 | ⭐⭐⭐ |
| **B. 语义搜索** | `discover_skills(query="图片放大")` → embedding 匹配 | 精准；省 token | 需要向量索引；冷启动问题 | ⭐⭐⭐⭐ |
| **C. 分类浏览** | `discover_skills(category="视频")` → 按类别返回 | 结构清晰；用户也能理解 | 需要预设分类体系 | ⭐⭐⭐⭐ |
| **D. 混合模式** | 无参数返回分类列表，有参数时搜索或过滤 | 最灵活 | 实现复杂 | ⭐⭐⭐⭐⭐ |

**方案 D 混合模式详解**：

```python
def discover_skills(query: str = None, category: str = None) -> dict:
    """
    发现 Passive Skills。

    - 无参数：返回分类概览
    - query：语义搜索
    - category：按类别过滤
    """

    if not query and not category:
        # 返回分类概览
        return {
            "categories": [
                {"name": "视频", "count": 5, "skills": ["generate_video", "video_edit", ...]},
                {"name": "图片", "count": 8, "skills": ["generate_image", "upscale", ...]},
                {"name": "音频", "count": 3, "skills": ["tts", "music_gen", ...]},
                {"name": "工具", "count": 10, "skills": ["ocr", "translate", ...]},
            ],
            "hint": "使用 discover_skills(category='视频') 查看详情，或 discover_skills(query='图片放大') 搜索"
        }

    if category:
        # 返回该类别下所有 Skills 的 Level 1
        return {
            "category": category,
            "skills": [
                {"name": "generate_video", "description": "AI 视频生成..."},
                {"name": "video_edit", "description": "视频剪辑处理..."},
            ]
        }

    if query:
        # 语义搜索，返回 Top-K 相关 Skills
        return {
            "query": query,
            "results": [
                {"name": "upscale", "description": "图片超分辨率放大...", "relevance": 0.95},
                {"name": "enhance", "description": "图片增强修复...", "relevance": 0.72},
            ]
        }
```

**分类体系建议**：

```yaml
skill_categories:
  生成:
    - generate_image
    - generate_video
    - generate_audio
    - generate_music
  处理:
    - ffmpeg
    - upscale

    - compress
  文本:
    - tts
    - stt
    - translate
    - ocr
  分析:
    - describe_image
    - analyze_video
    - extract_info
  工具:
    - web_fetch
    - web_search
    - browser_control
```

### 7.4 与 Studio 的联动

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **A. Studio 预设 Focus** | Studio 模板包含推荐的 Focus Profile | 开箱即用；场景优化 | 用户自由度低 |
| **B. Studio 不管 Focus** | Studio 只管 UI 和默认配置，Focus 独立 | 解耦清晰 | 用户每次都要 init |
| **C. Studio 建议 + 可覆盖** | Studio 提供推荐 Focus，用户可自行调整 | 平衡；最佳实践 | 实现稍复杂 |

**方案 C 实现**：

```yaml
# studio-templates/advertising-video/studio.yaml
name: "广告视频 Studio"
description: "面向广告短片制作的工作空间"

layout:
  # UI 配置...

config:
  # 默认配置...

recommended_focus:
  scenario: "广告视频制作"
  active:
    - generate_video
    - generate_image
    - storyboard
    - tts
    - ffmpeg
    - subtitle
  note: "这是推荐配置，你可以用 /init 调整"
```

用户使用 Studio 时：
1. 提示「此 Studio 推荐以下 Focus 配置，是否应用？」
2. 用户确认后应用
3. 用户也可以跳过，保持默认或自己 init

## 八、演化机制深化

### 8.1 自动演化触发条件

| 触发条件 | AGENT 行为 | 用户交互 |
|---------|-----------|---------|
| 某 Passive Skill 在 3 次对话中被手动发现使用 | 建议提升为 Active | "要把 X 加到核心能力吗？" |
| 某 Active Skill 连续 10 次对话未使用 | 建议降级为 Passive | "X 能力最近没用到，要移到后备吗？" |
| 用户任务类型明显变化 | 建议重新 init | "看起来你在做后期工作了，要调整一下焦点吗？" |
| 发现新 Skill（安装/更新后） | 提示用户 | "发现新能力 X，要加到核心列表吗？" |

### 8.2 演化记录

```yaml
# project/.cowork/focus.yaml
evolution_history:
  - date: "2025-02-01"
    action: "init"
    changes: ["+ generate_video", "+ storyboard", "- data_analysis"]
    trigger: "user_init"

  - date: "2025-02-03"
    action: "promote"
    changes: ["+ subtitle"]
    trigger: "frequent_use"
    note: "用户在 3 次对话中使用了字幕生成"

  - date: "2025-02-05"
    action: "demote"
    changes: ["- music_gen"]
    trigger: "inactive"
    note: "连续 10 次对话未使用"
```

### 8.3 用户控制演化行为

```yaml
# project/.cowork/focus.yaml
evolution_settings:
  auto_promote: true       # 自动提升频繁使用的 Skill
  auto_demote: false       # 不自动降级（需要用户确认）
  suggest_reinit: true     # 检测到场景变化时建议重新 init
  promote_threshold: 3     # 使用 N 次后建议提升
  demote_threshold: 10     # N 次未用后建议降级
```

## 九、完整命令设计

### 9.1 /init 命令族

| 命令 | 作用 | 示例 |
|------|------|------|
| `/init` | 引导式初始化（修改当前会话） | AGENT 开始引导对话 |
| `/init save` | 引导式初始化并保存到项目级 | 同上，但持久化 |

| `/init import <file>` | 从文件导入配置 | 应用他人的配置 |

### 10.2 discover_skills 工具定义

```yaml
name: discover_skills
description: |
```

```
└── .skills/
    └── ...                  # 项目级 Skills
```

### 验证方式

1. **Prompt 组装验证**：模拟一个有 30 个 Skills 的项目，对比 init 前后的系统 prompt token 数
2. **行为稳定性验证**：同一个任务，对比 Active 5 个 vs 全量 30 个 Skills 时 AGENT 的工具选择一致性
3. **引导流程验证**：模拟完整的 init 对话，确认 AGENT 能正确引导用户完成配置
4. **演化验证**：模拟使用场景，验证自动提升/降级建议的触发逻辑

## 十二、Skill 推荐机制设计

### 12.1 问题定义

/init 引导时，Agent 需要回答一个关键问题：**在用户描述的场景下，哪些 skills 应该 Active？**

推荐的质量直接决定了 /init 的价值——推错了，不如不推。

推荐面临三个挑战：

| 挑战 | 说明 | 示例 |
|------|------|------|
| **信息不对称** | Agent 只有 skill 的 name+description，判断依据太少 | "bash" 该不该推荐？取决于用户要不要做自动化 |
| **场景模糊** | 用户描述的场景可能不精确 | "做视频"——广告？Vlog？动画？差异巨大 |
| **干扰风险** | 某些 skill 组合会让 Agent 行为发散 | generate-video + browser-control 同时存在，Agent 可能走浏览器路径而非 API |

### 12.2 三层推荐策略

推荐不是单一机制，而是三层策略逐层兜底：

```
用户描述场景
    │
    ▼
Layer 1: 场景模板匹配（秒级，确定性最高）
    │ 命中 → 直接推荐模板配置
    │ 未命中 ↓
    ▼
Layer 2: Skill 元数据匹配（基于增强的 SKILL.md）
    │ Agent 根据 use_when / dont_use_when 逐一评估
    │ 信息不足 ↓
    ▼
Layer 3: LLM 结构化推理（兜底，最灵活但最不稳定）
    │ Agent 按评估框架自行判断
    ▼
输出推荐列表 → 干扰检查 → 展示给用户确认
```

**设计原则**：Layer 1 保证下限，Layer 2 提升精度，Layer 3 覆盖长尾。绝大多数场景应该在 Layer 1 就能解决。

### 12.3 Layer 1：场景模板

预定义高频场景的 skill 套餐。这是推荐质量的**下限保障**——即使 Agent 推理能力波动，模板也能给出靠谱的基础配置。

#### 模板格式

```yaml
# 系统内置 或 .cowork/scenario_templates.yaml
scenario_templates:

  广告视频:
    aliases: [产品视频, 带货视频, 投放素材, 种草视频, 商业短片, 千川素材]
    description: "面向投放的 15-60 秒产品广告"
    active:
      - name: storyboard
        role: primary
        reason: "广告需要精确的分镜控制"
      - name: generate-video
        role: primary
        reason: "核心生成能力"
      - name: generate-image
        role: primary
        reason: "产品素材、场景图"
      - name: tts
        role: primary
        reason: "配音旁白"
      - name: ffmpeg
        role: primary
        reason: "剪辑拼接、格式转换"
      - name: subtitle
        role: supporting
        reason: "广告字幕"
    excluded_note: "代码执行、浏览器控制、数据分析等能力已转为按需模式"

  漫画漫剧:
    aliases: [条漫, 漫画, 分镜漫画, 漫剧, 动态漫画, 有声漫画]
    description: "静态漫画或有声漫剧制作"
    active:
      - name: storyboard
        role: primary
        reason: "漫画的核心是分镜叙事"
      - name: generate-image
        role: primary
        reason: "画面生成"
      - name: image-edit
        role: primary
        reason: "构图调整、局部重绘、一致性维护"
      - name: subtitle
        role: supporting
        reason: "对话框/旁白文字"
      - name: tts
        role: supporting
        reason: "漫剧配音（仅动态漫剧需要）"

  有声内容:
    aliases: [播客, 有声书, 配音, 旁白, 音频节目, ASMR]
    description: "以音频为核心的内容制作"
    active:
      - name: tts
        role: primary
        reason: "核心语音合成"
      - name: music-gen
        role: primary
        reason: "背景音乐/BGM"
      - name: ffmpeg
        role: supporting
        reason: "音频处理、混音、格式转换"

  图文内容:
    aliases: [小红书, 公众号, 图文, 海报, 社交媒体图片, 封面]
    description: "图片+文案类社交媒体内容"
    active:
      - name: generate-image
        role: primary
        reason: "配图/素材生成"
      - name: image-edit
        role: supporting
        reason: "图片微调"
      - name: web-fetch
        role: supporting
        reason: "参考素材收集"
```

#### 模板匹配逻辑

Agent 不需要精确匹配模板名，而是基于 `aliases` 和 `description` 进行语义匹配：

```
1. 提取用户输入中的关键词
2. 与每个模板的 aliases 做关键词交集
3. 明确命中 1 个 → 直接推荐
4. 模糊命中多个 → 列出候选让用户选择
5. 全部未命中 → 进入 Layer 2
```

**关键设计**：模板不是死板的——Agent 在展示模板推荐后，应该基于用户的**具体描述**做微调。例如用户说"做抖音带货视频，需要 AI 配乐"，命中「广告视频」模板后，Agent 应主动建议把 `music-gen` 从 Passive 提升到 Active。

#### 用户可扩展

```yaml
# project/.cowork/custom_templates.yaml（用户自定义模板）
scenario_templates:
  我的日常广告流:
    aliases: [日常]
    description: "我常用的广告制作配置"
    active:
      - name: storyboard
        role: primary
      - name: generate-video
        role: primary
      # ...用户自己沉淀的配置
```

自定义模板优先级高于系统内置模板。

### 12.4 Layer 2：Skill 元数据增强

当模板匹配不上时，Agent 需要从 skill 自身的元数据来判断。当前 SKILL.md 只有 `name` + `description`，信息太稀薄。

#### 增强后的 SKILL.md 格式

在现有 `metadata.cowork` 下新增推荐相关字段：

```yaml
---
name: generate-video
description: "AI 视频生成。支持 Sora、Minimax、Runway 等平台。"
allowed-tools: VideoGeneration
metadata:
  cowork:
    # === 现有字段（不变）===
    category: 生成
    requires:
      bins: ["ffmpeg"]
      env: ["OPENAI_API_KEY"]
    icon: 🎬

    # === 新增：推荐相关字段 ===

    # 什么时候应该推荐/使用这个 skill
    use_when:
      - "用户需要从文本或图片生成视频片段"
      - "用户提到 Sora、Minimax、Runway、可灵等视频平台"
      - "用户需要 AI 生成的动态画面"

    # 什么时候不应该推荐/使用（防误推荐 + 运行时防误用）
    dont_use_when:
      - "用户想剪辑或拼接已有视频 → 应该用 ffmpeg"
      - "用户想录屏或截取视频片段 → 应该用 ffmpeg"
      - "用户想做实时视频流 → 超出能力范围"

    # 常见搭配（推荐时的连带效应）
    pairs_well_with: [storyboard, tts, ffmpeg, generate-image]

    # 容易混淆的 skill（干扰风险声明）
    confusable_with:
      - skill: browser-control
        confusion: "Agent 可能尝试用浏览器操作视频平台网页，而非调用 API"
        resolution: "优先使用 generate-video 的 API，仅当 API 不支持该平台时才考虑浏览器"
      - skill: bash
        confusion: "Agent 可能写脚本直接调用 ffmpeg 来'生成'视频"
        resolution: "ffmpeg 用于处理已有视频，AI 生成新视频应该用 generate-video"

    # 推荐时的默认角色
    typical_priority: primary  # primary / supporting / rare
    # 典型使用场景（用于 Layer 2 快速过滤）
    typical_scenarios: [广告视频, 短视频, 动画, 产品展示, MV]
---
```

#### 各字段在推荐流程中的作用

| 字段 | 推荐阶段 | 运行时 | 说明 |
|------|---------|--------|------|
| `use_when` | ✅ 正向推荐依据 | ✅ 调用决策参考 | Agent 看到匹配的信号就加分 |
| `dont_use_when` | ✅ 负向排除依据 | ✅ 防误用 | Agent 看到排除信号就减分/排除 |
| `pairs_well_with` | ✅ 连带推荐 | — | 推了 A 顺带考虑推 B |
| `confusable_with` | ✅ 干扰检查 | ✅ 路径选择 | AB 不该同时 Active，或需要明确优先级 |
| `typical_priority` | ✅ 默认角色 | — | 推荐为 primary 还是 supporting |
| `typical_scenarios` | ✅ 快速过滤 | — | 场景名关键词匹配 |

#### Layer 2 的评估流程

```
Agent 拿到所有可用 Skills 的增强元数据后：

1. 快速过滤：typical_scenarios 是否包含用户场景关键词
   → 命中的 skill 进入候选池

2. 正向评估：use_when 是否匹配用户描述
   → 匹配的加分

3. 负向排除：dont_use_when 是否匹配当前场景
   → 匹配的排除或降级

4. 连带扩展：候选池中的 skill 的 pairs_well_with 指向谁
   → 被多次指向的 skill 加分

5. 角色分配：根据 typical_priority 确定 primary/supporting

6. 干扰检查：见 12.6 节
```

### 12.5 Layer 3：LLM 结构化推理

当前两层都无法覆盖时（全新场景、跨领域需求），给 Agent 一个结构化的评估框架，让它按步骤判断而非凭感觉选。

#### 推理框架（注入 Agent prompt）

```markdown
## Skill 推荐评估框架（当模板和元数据不足时使用）

对每个可用 skill，完成以下评估：

### 1. 相关性判定
- 强相关：场景的核心环节必须用到 → 候选 Active (primary)
- 中相关：场景中可能用到但非必需 → 候选 Active (supporting)
- 弱相关：理论上能用但当前场景大概率不需要 → Passive
- 无关：完全无关 → Passive

### 2. 数量控制
- Active Skills 总数控制在 5-10 个
- 其中 primary 不超过 5 个（核心能力必须聚焦）
- 如果强相关的 skill 超过 10 个，按以下优先级裁剪：
  a. 保留与「输出物」直接相关的（生成、处理）
  b. 保留与「核心流程」相关的（规划、编排）
  c. 降级「辅助型」能力（搜索、分析）

### 3. 干扰检查
- 推荐列表中是否有功能重叠的 skill？
- 是否有「万能型」skill（如 bash）可能抢占专用 skill 的使用场景？
- 如果有，保留更专用的那个，降级更通用的那个

### 4. 生成推荐理由
每个推荐的 Active Skill 必须附一句话理由，说明「为什么这个场景需要它」
```

**设计意图**：这个框架不是给代码执行的，是给 Agent 的思维链（Chain-of-Thought）用的。通过结构化的评估步骤，减少 Agent 的随意性。

### 12.6 干扰规避机制

这是 /init 的核心价值所在——不只是"推荐哪些"，更是"排除哪些会造成干扰的"。

#### 干扰的三种类型

| 类型 | 描述 | 典型案例 | 后果 |
|------|------|---------|------|
| **能力混淆** | 两个 skill 能做类似的事，Agent 不知道选哪个 | `generate-video` vs `browser-control`：都能"生成视频"但路径不同 | Agent 随机选择，行为不稳定 |
| **越界使用** | 某 skill 能力范围太广，Agent 在不该用时用 | `bash` 什么都能做，Agent 可能绕过专用 skill | 绕过了 skill 的 gating、权限控制和最佳实践 |
| **路径干扰** | 多个 skill 组合出一条非最优路径 | `code-execution` + `web-fetch` → Agent 写爬虫而非用搜索 skill | 效率低、结果差、用户体验差 |

#### 干扰检查算法

在推荐列表确定后、展示给用户前，执行干扰检查：

```
for skill_a in recommended_active_skills:
    for conflict in skill_a.confusable_with:
        if conflict.skill in recommended_active_skills:
            # 发现干扰对
            → 根据 conflict.resolution 和当前场景判断保留哪个
            → 被排除的降级为 Passive
            → 在推荐展示中说明排除理由

# 额外检查：通用工具越界
for skill in recommended_active_skills:
    if skill.name in ["bash", "code-execution", "browser-control"]:
        # 通用工具需要 scope 限制
        → 在 Focus Context 中注入明确的使用边界
```

#### 运行时使用边界注入

即使通过了推荐阶段的干扰检查，运行时仍需要**使用边界**来防止误用：

```yaml
# Focus Context 中的使用边界声明（注入系统 prompt）
active:
  - name: generate-video
    role: primary
    scope: "AI 视频生成（文生视频、图生视频）"
    boundary: "剪辑/拼接已有视频请使用 ffmpeg，不要用此 skill"

  - name: ffmpeg
    role: primary
    scope: "视频处理（剪辑、拼接、格式转换、加字幕）"
    boundary: "生成新视频内容请使用 generate-video，不要用 ffmpeg"

  - name: bash
    role: supporting
    scope: "文件管理、简单脚本"
    boundary: "有专用 skill 可用时，优先使用专用 skill；不要用 bash 替代 generate-video / ffmpeg / tts 等能力"
```

**scope** 定义"该做什么"，**boundary** 定义"不该做什么"。两者结合形成清晰的使用边界。

### 12.7 增强版 Focus Context Prompt

结合推荐机制和干扰规避，更新第十章的 Focus Context 注入模板：

```markdown
## 当前场景聚焦

<focus_profile>
场景：{{scenario_description}}
匹配来源：{{matched_template_name or "自定义配置"}}
</focus_profile>

### 核心能力（优先使用）
{{#each active_primary}}
- **{{name}}**：{{scope}}
  {{#if boundary}}⚠️ {{boundary}}{{/if}}
{{/each}}

### 辅助能力（按需使用）
{{#each active_supporting}}
- {{name}}：{{scope}}
  {{#if boundary}}⚠️ {{boundary}}{{/if}}
{{/each}}

### 行为规则
1. 完成任务时，**优先从核心能力中选择工具**
2. 核心能力不够时使用辅助能力
3. 以上都无法满足时，使用 `discover_skills` 探索更多能力
4. **严格遵守每个能力的使用边界**——遇到边界提示的情况时，切换到正确的工具
5. 不要用通用工具（bash、code-execution）替代专用能力
6. 如果反复需要某个 Passive Skill，建议用户 `/init add` 将其加入核心
```

## 十三、增强版引导流程

整合推荐机制后，更新第三章的引导流程。

### 完整 /init 流程

```
Step 1: 场景识别
├── AGENT: "你这次主要想做什么？"
├── 用户: 描述需求
├── AGENT 内部: 提取关键信息（创作类型、场景、平台、特殊需求）
│
Step 2: 模板匹配（Layer 1）
├── 命中 1 个模板 → 展示推荐，跳到 Step 4
├── 模糊命中多个 → 列出候选让用户选
├── 未命中 → 进入 Step 3
│
Step 3: 元数据 + 推理评估（Layer 2 + 3）
├── AGENT 内部: 基于 skill 元数据的 use_when / typical_scenarios 评估
├── 不足时使用结构化推理框架
├── 生成推荐列表
│
Step 4: 干扰检查
├── AGENT 内部: 遍历 confusable_with 声明
├── 发现冲突 → 保留更合适的，降级另一个
├── 通用工具 → 添加 scope + boundary
│
Step 5: 展示推荐 & 用户确认
├── 展示 primary + supporting 分组
├── 每个 skill 附推荐理由
├── 说明被排除的 skill 及原因
├── 用户: 确认 / 增删调整
│
Step 6: 应用配置
├── 生成 Focus Profile（含 scope + boundary）
├── 注入 Focus Context 到系统 prompt
├── 提示：随时 /init 调整
```

3. **演化建议的展示时机？** 是在对话结束时汇总，还是即时提示？
4. **多用户/团队场景下的 Focus 共享？** 是否支持团队级别的 Focus 模板？
5. **场景模板的冷启动**：初始版本内置多少个模板？如何收集用户反馈来迭代模板？
6. **confusable_with 的维护成本**：skill 数量增长后，两两组合的干扰声明是否会爆炸？是否需要自动化检测？
7. **推荐准确率的度量**：如何量化推荐质量？用户修改推荐的比例是否可以作为指标？

## 十五、与现有 Claude Code Skill 机制的对比

| 维度 | Claude Code | Creative CoWork /init |
|------|-------------|----------------------|
| Skill 激活方式 | 用户通过 /skill 显式调用 | Active Skills 自动可用，Passive 需发现 |
| 聚焦机制 | 无（所有 Skills 平等） | 有（Active/Passive 分层 + scope/boundary） |
| 推荐机制 | 无 | 三层策略（模板 → 元数据 → LLM 推理） |
| 干扰规避 | 无 | confusable_with 声明 + 运行时边界 |
| 演化 | 无 | 有（基于使用模式自动/手动调整） |
| 持久化 | Skill 配置持久化 | Focus Profile 持久化 |
| 场景适配 | 通用开发场景 | 创作场景优化 |

**核心差异**：Claude Code 的 Skills 是"工具扩展"，Creative CoWork 的 /init 是"注意力管理"。前者解决"能做什么"，后者解决"**优先做什么、怎么做、不该怎么做**"。
