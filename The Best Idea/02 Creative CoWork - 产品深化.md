# Creative CoWork - 产品深化

> 基于 DEMO 思路的扩展讨论和关键决策

---

## 一、核心定义

### 产品一句话描述

**Creative CoWork = 创作项目的上下文容器 + AGENT + 可定制的 Studio**

### 类比

- 程序员用 VS Code 管理代码项目（代码、配置、终端、Git 都在一个窗口）
- 超创需要一个东西管理创作项目（素材、生成结果、参考图、对话历史都在一个空间）
- 更直接的类比：**Claude Code 的创作版**，或者 **Kuse**

### 与现有产品的本质区别

> "在创作领域深度考虑上下文之后的 AI native 产品"

| 产品 | 定位 | 缺失的部分 |
|------|------|-----------|
| ComfyUI | 节点编排工具 | 不考虑项目级上下文 |
| Runway | 生成工具 | 不考虑项目级上下文 |
| Descript | 编辑工具 | 上下文有限，能力不够开放 |
| **Creative CoWork** | 上下文容器 + AGENT | - |

---

## 二、目标用户：超创

### 定义

超创是目前已经能够用 AIGC 生成赚到钱的小团队或个人。

### 具体例子

- 5 人团队，1 个月内用 AIGC 交付伊利牛奶 7 集广告短片
- 1 个人，3 个月用 AIGC 工具做 10 分钟短片入选戛纳电影节

### 超创的核心特点

1. **会自己制作和改造工具**
2. **能够靠创作赚钱或深度表达**
3. 与普通用户的区别：普通用户以非盈利的体验为主

### 超创的核心痛点

> **在创作流程中的上下文太分散，导致创作效率太低。**

不是能力不够，不是工具不好，而是：素材、参考、中间产物、配置散落在 10 个地方，来回切换太累。

---

## 三、核心概念

### 上下文（Context）

创作项目的上下文包括但不限于：

| 类型 | 说明 |
|------|------|
| 素材 | 图、视频、音频、文本 |
| 生成结果 | 包括失败的和成功的 |
| 参考/灵感 | 情绪板、参考视频 |
| 对话历史 | 和 AGENT 的交互记录 |
| 配置/偏好 | 常用的模型、参数、风格词 |
| 版本/分支 | 不同方向的尝试 |

### Studio

**Studio = 场景化的工作空间模板**

包含：
- 预设的 UI 布局
- 预装的 Skills
- 适合该场景的默认配置

例子：
- **MV Studio**：音乐轨道 + 分镜面板 + 视频生成能力
- **广告片 Studio**：脚本编辑器 + 品牌素材库 + 多版本对比
- **个人短片 Studio**：情绪板 + 角色设定 + 连贯性管理

### 用户使用路径

**路径 A：基于成熟 Studio**
1. 选一个接近自己需求的 Studio
2. 基于自己的项目开始工作
3. 一边工作一边小修小改
4. 改过的 Studio 变成自己的资产

**路径 B：从零开始**
1. 从空白开始
2. 一边做项目一边搭建自己的 Studio
3. Studio 随项目逐步成型

### 自然语言 + UI 的分工

| 组件 | 角色 |
|------|------|
| 自然语言 | 指挥 AGENT |
| UI | 展示素材，基于多模态表达 |

### GENUI 的角色

GENUI 在产品中承担多重角色：
- 根据用户的项目类型，自动生成合适的 UI 布局
- 根据 Skill 的输入输出，自动生成操作界面
- 根据对话内容，动态生成临时的交互组件

## 四、关键判断

### 时间窗口

- 未来 1-2 年内会成立
- 现在开始可以试探入场

### 为什么是现在？

之前技术成熟度不够，现在够了，特别是：
- **Skills**：能力模块化的技术成熟
- **GENUI**：界面自动生成的技术成熟

### 做成这件事最关键的要素

1. 多模态上下文的优化、生成能力的进一步强化
2. 和超创的联系（理解需求、获取反馈）

### 最大的风险

**合规风险**——开放接入各种能力可能带来的合规问题

## 五、核心问题讨论

### 问题 1：核心赌注的前提条件，哪个风险最高？

赌注：**"上下文聚合 + AGENT"的组合，能让超创的效率产生质变**

前提条件：
- A. AGENT 足够聪明，能理解创作上下文并做出有用的动作
- B. 上下文的聚合本身能产生"1+1>2"的效果
- C. 超创愿意把工作流迁移到一个新平台

**结论**：A 是最高的，B 和 C 基本已经在 coding 领域看到了例子

---

### 问题 2：Studio 的冷启动

- 最早的 Studio 谁来做？
- 没有 Studio 生态时，早期用户的体验如何保证？
- Studio 的分享/交易机制怎么设计才能激励贡献？

**结论**：
- 内部团队做，或者 AGENT 来做
- 可能需要强客服
- 逐级积分分润制

---

### 问题 3：开放性 vs 平台价值

用户可以自由选择 Skill，甚至接入外部的、竞品的能力。

- 如果用户可以接入任何能力，平台的价值在哪？
- 如果最好的能力都在外部，用户为什么不直接用那些工具？
- 怎么平衡"开放带来的吸引力"和"平台自身的价值锚点"？

**结论**：
- 平台的价值在于维持了长时序的上下文，做好上下文，用户粘性就会很大
- 上下文是割裂的
- 开放能力，集中上下文，这个问题应该不成立了

---

### 问题 4：超创的粘性来源

超创有能力自己搭建工具，也有能力离开平台自己搞一套。

平台对他们的粘性来自哪里？
- A. 节省他们自己搭建的时间（效率）
- B. 提供他们自己搞不定的能力（能力）
- C. 提供一个社区/生态，可以和其他超创交流、交易（网络）
- D. 其他

**结论**：
- A 是最成立的
- B、C 一般成立

---

### 问题 5：如果只做一个场景的 Studio 来验证

资源有限，先做哪个？
- MV/音乐可视化
- 广告短片
- 个人艺术短片
- 短视频/自媒体内容
- 其他

选择的依据是什么？

**结论**：
- 广告：真正能赚钱的内容

---

### 问题 6：AGENT "足够聪明"的标准是什么？

AGENT 需要做到什么程度才算"够用"？

- 能理解"把这个视频的风格换成赛博朋克"？
- 能理解"参考这张图的构图，用那段文案的情绪"？
- 能自己判断"这个生成结果不够好，我换个参数再试"？
- 还是更高的要求？

**结论**：只要能够尽可能地利用已有能力去完成任务即可。目前的智能程度，似乎已经可以了。

**扩展分析**：

AGENT 不需要"创造"能力，只需要"编排"能力。类比 Claude Code：Claude 不会发明新的编程语言，但它能理解意图，用现有的语言、库、工具帮你完成任务。

**最小可用智能的四个层次：**

| 层次 | 描述 | 示例 |
|------|------|------|
| Level 1 | 指令执行 | "把这张图改成 1080p" → 调用 Skill |
| Level 2 | 意图理解 | "这张图有点糊" → 理解为需要超分 |
| Level 3 | 上下文利用 | "用之前那个风格" → 从上下文找到风格配置 |
| Level 4 | 多步编排 | "做成抖音版本" → 裁剪 + 字幕 + 调时长 |

**MVP 底线：Level 3**（上下文利用），Level 4 也需要完成。

**关键洞察：瓶颈不在模型智能，而在于上下文能否被有效结构化。** 这是产品/工程问题，不是模型能力问题。

**补充：不害怕人工介入**——在创作领域，human in the loop 往往是高效的，因为创作的"好坏"是主观的，需要人来判断。AGENT 负责执行和编排，人负责审美和决策。

---

### 问题 7：上下文的"长时序"怎么理解？

"维持长时序的上下文"，这个"长"是指：

- A. 单个项目内的上下文（几天到几周）
- B. 跨项目的上下文（这个用户的风格偏好、常用配置）
- C. 跨用户的上下文（某个 Studio 被很多人用，积累了集体智慧）

哪个是最重要的？

**结论**：MVP 版本是 A，长期来看都有

---

### 问题 8：广告场景的具体拆解

广告的创作流程大概是：

```
Brief → 创意/脚本 → 分镜 → 素材生成 → 剪辑合成 → 交付
```

打算覆盖全流程，还是先切其中一段？如果切一段，切哪段？

**结论**：必须是全流程覆盖，不然仍然会出现上下文隔离的问题

---

### 问题 9：和现有超创的关系

"和超创的联系"是关键要素之一。

现在和超创的联系是什么状态？
- 已经有深度合作的超创？
- 有认识但还没深度合作？
- 还需要建立联系？

如果要找 3-5 个超创做早期用户，知道去哪找、怎么说服他们吗？

**结论**：有很多关系很好的导演朋友

## 六、讨论记录

### Round 1：核心定义对齐

- 明确了产品本质：上下文容器 + AGENT + Studio
- 明确了目标用户：能用 AIGC 赚钱的超创
- 明确了核心痛点：上下文分散导致效率低

### Round 2：关键判断

- 核心风险是 AGENT 能力（技术风险 > 市场风险）
- 护城河是上下文积累，不是能力独占
- 粘性来自效率，不是能力壁垒
- 先切广告场景，因为用户在赚钱

### Round 3：深入细节

- **AGENT 智能标准**：Level 3（上下文利用）+ Level 4（多步编排）是 MVP 目标
- **核心瓶颈重新定义**：不是模型不够聪明，而是上下文能否被有效结构化（产品/工程问题）
- **Human in the loop**：在创作领域是高效的，不是障碍。AGENT 负责执行编排，人负责审美决策
- **上下文时序**：MVP 聚焦单项目内（A），长期覆盖 B 和 C
- **广告流程**：必须全流程覆盖，否则上下文仍然隔离
- **超创关系**：已有导演朋友资源，可以快速找到早期用户

---

## 七、核心结论汇总

### 护城河公式

```
护城河 = 上下文积累 × 时间
```

能力可以开放，用户可以接任何 Skill，但**上下文在这里积累**，迁移成本随时间增长。

### 风险分布

```
技术风险（AGENT 能力）> 市场风险（需求验证）
```

B（上下文聚合的价值）和 C（用户迁移意愿）已经被 coding 领域验证了。核心赌注落在 A：AGENT 能不能足够聪明。

### 切入策略

```
广告场景 → 因为用户真的在赚钱 → 付费意愿强 → 价值容易验证
```

## 八、调研问题清单

### 关于工作流和上下文痛点

1. 做一个项目时，素材、参考、中间产物都放在哪？几个工具/文件夹之间切换？
2. 有没有遇到过"找不到之前那个版本/那张参考图"的情况？怎么解决的？
3. 团队协作时，怎么同步项目进度和素材？用什么工具？

### 关于现有工具

4. 现在主要用哪些 AI 生成工具？各自用来做什么？
5. 这些工具之间怎么衔接？有没有特别烦的地方？
6. 有没有自己搭过一些自动化流程/工具？比如 ComfyUI workflow、脚本等

### 关于效率和瓶颈

7. 做一个项目，时间主要花在哪？（创意构思 / 素材生成 / 筛选调整 / 剪辑合成 / 其他）
8. 如果有个工具能帮你省时间，你最希望它省在哪个环节？

### 关于"理想工具"

9. 如果有一个地方能把所有素材、生成、参考、对话历史都放在一起，你会用吗？有什么顾虑？
10. 你会愿意把自己摸索出来的工作流/配置分享给别人吗？什么条件下愿意？
