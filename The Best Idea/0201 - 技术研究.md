## 一、OpenClaw/Moltbot 项目分析

### 1.1 项目概述

| 项目 | 信息 |
|------|------|
| 当前名称 | OpenClaw（原 Clawdbot → Moltbot） |
| GitHub | [moltbot/moltbot](https://github.com/moltbot/moltbot) |
| Stars | 109k+ |
| 定位 | 自托管的个人 AI 助手 |
| 作者 | Peter Steinberger (@steipete)，PSPDFKit 创始人 |

**核心理念**：能动手的 Claude —— 不只是聊天，而是能实际执行任务的 AI Agent。

### 1.2 核心功能

- **多渠道接入**：WhatsApp、Telegram、Slack、Discord、iMessage、Signal、Teams
- **语音交互**：macOS/iOS/Android 支持 always-on 语音唤醒
- **浏览器控制**：通过专用 Chrome 实例操作网页（CDP + Playwright）
- **设备集成**：本地命令执行、摄像头、屏幕录制
- **Skills 扩展**：模块化能力加载（700+ 社区 Skills）
- **多 Agent 路由**：不同渠道可路由到不同专属 Agent

### 1.4 与 Creative CoWork 的关系

- OpenClaw 偏向「通用个人助手」
- Creative CoWork 聚焦于「创作工作台」

## 二、Skills 架构对比

### 2.1 Anthropic Agent Skills 规范

2025年12月，Anthropic 发布 Agent Skills 作为开放标准，已被 27+ 工具采用（Claude Code、Cursor、VS Code、Codex、Gemini CLI 等）。

**官方资源**：
- 规范站点：[agentskills.io](https://agentskills.io)
- 官方仓库：[anthropics/skills](https://github.com/anthropics/skills)
- Claude Code 文档：[code.claude.com/docs/en/skills](https://code.claude.com/docs/en/skills)

#### 目录结构

```
skill-name/
├── SKILL.md          # 必需
├── scripts/          # 可选：可执行代码
├── references/       # 可选：参考文档
└── assets/           # 可选：模板、资源
```

#### SKILL.md 格式

```yaml
---
name: skill-name
description: 技能描述，用于 Agent 判断何时使用
license: Apache-2.0                    # 可选
compatibility: Requires git, docker    # 可选
metadata:                              # 可选
  author: example-org
  version: "1.0"
allowed-tools: Bash(git:*) Read        # 可选，实验性
---

# 技能标题

这里是 Markdown 指令内容...
```

#### Progressive Disclosure（渐进式披露）

```
Level 1: Metadata (~100 tokens)
  - name + description
  - 启动时加载到系统 prompt
      ↓
Level 2: Instructions (< 5000 tokens)
  - SKILL.md 完整内容
  - 激活时加载
      ↓
Level 3+: Resources (按需)
  - scripts/, references/, assets/
  - 执行时按需加载
```

### 2.2 Claude Code 扩展特性

Claude Code 在标准之上增加了以下能力：

| 字段 | 说明 |
|------|------|
| `disable-model-invocation` | `true` 时仅用户可调用，Claude 不会自动触发 |
| `user-invocable` | `false` 时隐藏于 `/` 菜单，仅 Claude 可调用 |
| `context` | 设为 `fork` 在独立 Subagent 中运行 |
| `agent` | 指定 Subagent 类型（Explore、Plan 等） |
| `model` | 指定使用的模型 |
| `hooks` | Skill 级别的生命周期钩子 |
| `argument-hint` | 自动补全时的参数提示 |

**Shell 预处理**：

```yaml
---
name: pr-summary
context: fork
agent: Explore
---

## PR Context
- PR diff: !`gh pr diff`        # 命令在注入前执行
- PR comments: !`gh pr view --comments`
```

**变量替换**：
- `$ARGUMENTS` - 所有参数
- `$0`, `$1` - 按位置访问参数
- `${CLAUDE_SESSION_ID}` - 会话 ID

### 2.3 OpenClaw 扩展特性

OpenClaw 在标准之上增加了 **Gating 系统**：

```json
{
  "openclaw": {
    "requires": {
      "bins": ["ffmpeg"],        // 必须存在的二进制
      "anyBins": ["chrome"],     // 至少存在其一
      "env": ["API_KEY"],        // 必须设置的环境变量
      "config": ["models.openai"] // 必须存在的配置
    },
    "os": ["darwin", "linux"],   // 支持的操作系统
    "primaryEnv": "API_KEY",     // 主要密钥变量
    "always": true               // 始终加载
  }
}
```

**Token 成本计算**：
```
基础开销: 195 chars（当 ≥1 skill）
每个 Skill: 97 + name_len + description_len + location_len
粗估: ~24 tokens/skill
```

### 2.4 对比总结

| 特性 | Anthropic 标准 | Claude Code | OpenClaw |
|------|---------------|-------------|----------|
| 跨平台兼容 | ✅ 27+ 工具 | ✅ | ❌ 仅自家 |
| Subagent 执行 | - | ✅ `context: fork` | ❌ |
| Shell 预处理 | - | ✅ `!command` | ❌ |
| Gating 系统 | ❌ | ❌ | ✅ 完整 |
| Hooks | - | ✅ | ❌ |
| 市场生态 | 分散 | Plugin 系统 | ClawdHub |

### 2.5 Creative CoWork 建议

1. **遵循 Anthropic Agent Skills 标准**：保证与 Claude Code 等工具兼容
2. **借鉴 OpenClaw Gating 系统**：创作场景有大量外部依赖
3. **聚焦创作场景的 Skill 类型**：风格迁移、分镜生成、多模态输出

## 三、浏览器控制原理

### 3.1 核心技术：Chrome DevTools Protocol (CDP)

CDP 是 Chrome 浏览器暴露的调试协议，允许外部程序：
- 打开/关闭标签页
- 导航到 URL
- 执行 JavaScript
- 截屏
- 监听网络请求
- 模拟用户输入

```bash
# Chrome 启动时开启远程调试端口
chrome --remote-debugging-port=9222

# 通过 WebSocket 连接
ws://127.0.0.1:9222/devtools/browser/<id>
```

### 3.2 OpenClaw 浏览器架构

```
┌──────────────────────────────────────────────────────────────────┐
│              Browser Control Server (HTTP)                       │
│                 127.0.0.1:18791 (loopback only)                  │
└──────────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
     ┌─────────────┐  ┌─────────────┐  ┌─────────────┐
     │   CDP 协议   │  │  Playwright │  │  Extension  │
     │  (基础通信)  │  │  (高级操作)  │  │   Relay    │
     └─────────────┘  └─────────────┘  └─────────────┘
              │               │               │
              ▼               ▼               ▼
     ┌─────────────────────────────────────────────────┐
     │     Chromium-based Browser (隔离实例)            │
     └─────────────────────────────────────────────────┘
```

### 3.3 三种 Profile 模式

| 模式 | 说明 | 使用场景 |
|------|------|---------|
| **openclaw-managed** | 独立 Chromium 实例，独立用户数据目录 | 默认，完全隔离 |
| **remote CDP** | 连接远程浏览器的 CDP 端口 | 服务器部署 |
| **extension relay** | 通过 Chrome 扩展控制现有浏览器 | 复用已登录 session |

### 3.4 Snapshot 机制

把页面结构转成文本，让 LLM 能"看懂"页面：

**AI Snapshot（数字 ref）**：
```
Page: Google Search
- [1] textbox "Search"
- [2] button "Google Search"
- [3] button "I'm Feeling Lucky"
```

Agent 说 `browser.click 2` 即可点击按钮。

**关键**：Ref 在页面导航后会失效，需要重新 snapshot。

### 3.5 Claude Code Chrome 集成

Claude Code 使用 Chrome Extension + Native Messaging：

```
┌─────────────────────────────────────────┐
│         Claude Code CLI                 │
└─────────────────────────────────────────┘
                    │
          Native Messaging Protocol
                    │
                    ▼
┌─────────────────────────────────────────┐
│     Claude in Chrome Extension          │
│  (Content Scripts + Service Worker)     │
└─────────────────────────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│     User's Chrome (共享登录态)           │
└─────────────────────────────────────────┘
```

**关键能力**：
- 共享用户的登录状态
- 可操作 Google Docs、Gmail、Notion 等已登录应用
- 操作可见，用户可随时中断

### 4.3 路径一：API 调用（推荐 80% 场景）

```python
# video_generators.py
class SoraGenerator:
    async def generate(self, prompt: str, settings: dict):
        response = self.client.videos.create(
            model=settings.get("model", "sora-2"),
            prompt=prompt,
            duration=settings.get("duration", 4),  # 4, 8, 12 秒
            resolution=settings.get("resolution", "1280x720"),
        )
        # 轮询等待完成...
        return download_url

class MinimaxGenerator:
    async def generate(self, prompt: str, settings: dict):
        result = await fal_client.subscribe_async(
            "fal-ai/minimax/video-01",
            arguments={
                "prompt": prompt,
                "aspect_ratio": settings.get("aspect_ratio", "16:9"),
            }
        )
        return result["video"]["url"]
```

- 平台没有 API（如某些新功能）
- 用户订阅了网页版但不想付 API 费
- 需要使用网页版独有的功能

| Browser-Use 库 | ✅ | 低 | ⭐⭐⭐⭐ |

## 五、关键结论

### 5.1 Skills 系统

- 遵循 **Anthropic Agent Skills 标准**（SKILL.md + YAML frontmatter）
- 借鉴 **OpenClaw Gating 系统**做依赖检测
- 聚焦创作场景的能力模块

### 5.2 视频生成工具

- **API 优先**：主流平台都有 API，更稳定、更快、成本更低
- **浏览器是 Fallback**：仅用于无 API 的场景
- 实现 **Smart Router** 自动选择最佳路径

### 5.3 浏览器控制

- 核心技术：**CDP + Playwright**
- 复用用户登录态：**Chrome Extension + Native Messaging**
- 安全原则：操作可见、用户可中断、明确授权

## 六、Agent 上下文管理方法

> 核心问题：如何在有限的 Context Window 中高效利用信息？

### 6.1 问题背景：Context 爆炸

现代 Agent 系统面临的核心矛盾：

```
┌─────────────────────────────────────────────────────────────┐
│                    Context Window 容量                       │
│                      (128K - 200K tokens)                   │
└─────────────────────────────────────────────────────────────┘
                              ▲
                              │ 竞争
     ┌────────────────────────┼────────────────────────────┐
     │                        │                            │
┌────┴────┐            ┌──────┴──────┐            ┌────────┴───────┐
│ 系统指令 │            │   工具描述   │            │  对话历史/素材  │
│ ~5K     │            │   ~20K+     │            │    ~100K+      │
└─────────┘            └─────────────┘            └────────────────┘
```

**典型问题**：

| 场景 | 问题 |
|------|------|
| MCP 工具过多 | 每个工具都有详细描述，20 个工具就占用 10K+ tokens |
| Skills 全量加载 | 所有 Skill 文档一次性注入，context 迅速耗尽 |
| 长对话累积 | 历史消息线性增长，挤压有效信息空间 |
| 多素材项目 | 图片描述、文档内容、参考资料叠加 |

**本质矛盾**：Agent 能力越强（工具越多），单次可用信息空间越小。

---

### 6.2 核心方法：渐进式披露 (Progressive Disclosure)

#### 6.2.1 概念定义

Anthropic 官方定义：

> Progressive disclosure is the core design principle that makes Agent Skills flexible and scalable. Like a well-organized manual that starts with a table of contents, then specific chapters, and finally a detailed appendix, skills let Claude load information only as needed.

**核心思想**：像手册一样分层组织信息——目录 → 章节 → 详细附录，Agent 只在需要时加载相关内容。

#### 6.2.2 三层披露模型

```
┌─────────────────────────────────────────────────────────────┐
│  Level 1: 元数据层 (~100 tokens)                            │
│  - Skill 名称 + 一句话描述                                   │
│  - 始终加载在系统 prompt                                     │
│  - 作用：让 Agent 知道"有这个能力"                           │
└─────────────────────────────────────────────────────────────┘
                              │
                         用户触发激活
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Level 2: 指令层 (< 5000 tokens)                            │
│  - SKILL.md 完整内容                                        │
│  - 激活时注入                                                │
│  - 作用：告诉 Agent "怎么做"                                 │
└─────────────────────────────────────────────────────────────┘
                              │
                         执行时按需
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Level 3+: 资源层 (无上限)                                   │
│  - scripts/, references/, assets/                           │
│  - 执行时按需读取                                            │
│  - 作用：提供具体实现和参考                                   │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：有文件系统和代码执行能力的 Agent，不需要把所有信息都放进 context window。

#### 6.2.3 CLI 风格实现（MCP 替代方案）

来源：[musistudio/claude-code-router](https://github.com/musistudio/claude-code-router/blob/main/blog/en/progressive-disclosure-of-agent-tools-from-the-perspective-of-cli-tool-style.md)

**核心观点**：CLI 工具的 `--help` 模式天然就是渐进式披露。

**传统 MCP 问题**：
```json
// 每个功能都是独立工具，全部暴露
{
  "tools": [
    {"name": "pdf_extract_text", "description": "Extract text from PDF..."},
    {"name": "pdf_extract_tables", "description": "Extract tables from PDF..."},
    {"name": "pdf_merge", "description": "Merge multiple PDFs..."},
    {"name": "pdf_split", "description": "Split PDF into pages..."},
    {"name": "pdf_fill_form", "description": "Fill PDF form fields..."},
    // ... 更多工具，每个都有详细描述
  ]
}
// 问题：10 个 PDF 工具就占用 2000+ tokens
```

**CLI 风格 MCP**：
```json
// 只暴露一个工具入口
{
  "name": "pdf",
  "description": "PDF manipulation toolkit. Pass --help to see usage.",
  "input_schema": {
    "properties": {
      "argv": {
        "type": "array",
        "items": {"type": "string"}
      }
    }
  }
}
// 只占用 ~100 tokens
```

**调用流程示例**：

```
用户: "把 test.pdf 的文字提取出来"

Agent: pdf(["--help"])
返回:  pdf - Comprehensive PDF manipulation toolkit

       Domains:
         forms      Fill, extract, and validate PDF form fields
         documents  Create, merge, and split PDFs
         text       Extract text content from PDFs
         tables     Extract tables from PDFs
         images     Render PDF pages to PNG images

       Run `pdf <domain> --help` for domain help.

Agent: pdf(["text", "--help"])
返回:  text - Extract text content from PDFs

       Commands:
         extract  Extract text content from a PDF

       Run `pdf text <command> --help` for details.

Agent: pdf(["text", "extract", "--help"])
返回:  usage: pdf text extract --pdf PDF_PATH [--pages PAGES] [--out OUTPUT]

       options:
         --pdf PDF_PATH    Input PDF file
         --pages PAGES     Page selection like 1,3-5 or 'all'
         --out OUTPUT      Write extracted text to file

Agent: pdf(["text", "extract", "--pdf", "test.pdf", "--out", "result.txt"])
返回:  {"result": {"stdout": "Extracted 6 pages", "output_path": "result.txt"}}
```

**对比总结**：

| 维度 | 传统 MCP | CLI 风格 MCP |
|------|---------|-------------|
| 初始 context 占用 | 高（所有工具描述） | 低（仅入口描述） |
| 调用次数 | 1 次 | 3-4 次（含 help 查询） |
| 信息获取 | 一次性全部 | 按需逐层 |
| 适合场景 | 工具少、使用频繁 | 工具多、偶尔使用 |

### 6.3 相关方法：上下文优化策略

#### 6.3.1 上下文压缩 (Context Compression)

**原理**：用更少的 tokens 表达相同信息。

```
┌─────────────────────────────────────────────────────────────┐
│  原始对话                                                    │
│  User: 帮我写一个登录页面，要有用户名密码输入框，还要有...      │
│  Agent: 好的，我来帮你创建登录页面。首先我需要了解...          │
│  User: 用 React，样式用 Tailwind...                         │
│  ... (20 轮对话，10K tokens)                                │
└─────────────────────────────────────────────────────────────┘
                              │
                         压缩/摘要
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  压缩后                                                      │
│  [任务] 创建 React 登录页面                                   │
│  [技术栈] React + Tailwind + TypeScript                      │
│  [已完成] 基础布局、表单组件                                   │
│  [待完成] 表单验证、提交逻辑                                   │
│  (~200 tokens)                                               │
└─────────────────────────────────────────────────────────────┘
```

**实现方式**：

| 方式 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| LLM 摘要 | 用模型总结历史对话 | 语义保真度高 | 有延迟、有成本 |
| 规则提取 | 提取关键实体和决策 | 快速、可控 | 可能丢失上下文 |
| 滑动窗口 | 只保留最近 N 轮 | 简单 | 丢失早期关键信息 |
| 混合策略 | 摘要 + 最近 N 轮 | 平衡 | 实现复杂 |

#### 6.3.2 记忆系统 (Memory System)

**原理**：将信息外部化存储，按需检索。

```
┌─────────────────────────────────────────────────────────────┐
│                        Agent Core                            │
└─────────────────────────────────────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        ▼                     ▼                     ▼
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  短期记忆     │      │  长期记忆     │      │  工作记忆     │
│  (对话历史)   │      │  (持久存储)   │      │  (当前任务)   │
│              │      │              │      │              │
│  - 最近对话   │      │  - 用户偏好   │      │  - 任务目标   │
│  - 临时状态   │      │  - 历史决策   │      │  - 中间结果   │
│              │      │  - 项目知识   │      │  - 待办事项   │
└──────────────┘      └──────────────┘      └──────────────┘
```

**典型实现**：

- **Claude Code**：`CLAUDE.md` 作为项目级长期记忆
- **Cursor**：`.cursorrules` 存储项目规则
- **ChatGPT**：Memory 功能存储用户偏好
- **OpenClaw**：SQLite 存储对话历史和用户配置

#### 6.3.3 检索增强生成 (RAG)

**原理**：不把所有信息放进 context，而是根据当前问题检索相关内容。

```
用户问题: "登录失败怎么处理？"
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  向量检索                                                    │
│  Query → Embedding → 相似度搜索 → Top-K 结果                 │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  检索结果（只注入相关内容）                                    │
│  - auth/login.ts: handleLoginError() 函数                   │
│  - docs/error-handling.md: 错误处理规范                       │
│  - tests/auth.test.ts: 登录失败测试用例                       │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  Context = 系统指令 + 检索结果 + 用户问题                     │
│  (精准、紧凑、相关)                                           │
└─────────────────────────────────────────────────────────────┘
```

**适用场景**：

| 场景 | 说明 |
|------|------|
| 大型代码库 | 不可能全部放进 context，按需检索 |
| 知识库问答 | 文档太多，检索相关章节 |
| 多项目切换 | 每个项目的上下文独立管理 |

#### 6.3.4 工具路由 (Tool Routing)

**原理**：不暴露所有工具，根据意图动态加载。

```
用户: "帮我生成一个视频"
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  意图识别层                                                  │
│  输入 → 分类 → 视频生成类                                    │
└─────────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────────────────────────────────────────────────┐
│  动态加载相关工具                                             │
│  - sora_generate                                            │
│  - minimax_generate                                         │
│  - runway_generate                                          │
│  (其他 50 个无关工具不加载)                                   │
└─────────────────────────────────────────────────────────────┘
```

**实现方式**：

- **OpenAI Assistants**：通过 function calling 的 `tool_choice` 参数
- **LangChain**：ToolSelector 根据 query 选择工具子集
- **Claude Code**：Skills 的激活机制（用户通过 `/skill` 触发）

### 6.7 Anthropic 官方博客总结

> 来源：[Equipping Agents for the Real World with Agent Skills](https://claude.com/blog/equipping-agents-for-the-real-world-with-agent-skills)
> 发布日期：2025 年 10 月 16 日

#### 6.7.1 什么是 Agent Skills

Agent Skills 是 **"组织好的指令、脚本和资源文件夹，Agent 可以动态发现和加载，以更好地执行特定任务"**。

核心类比：就像给新员工写入职手册——把领域知识打包成可复用的模块，让通用 Agent 变成特定领域专家。

#### 6.7.2 三层渐进式架构

```
┌─────────────────────────────────────────────────────────────┐
│  Level 1: 元数据层                                           │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ SKILL.md (YAML frontmatter)                             ││
│  │ - name: pdf                                             ││
│  │ - description: PDF 处理工具，支持表单填写、文本提取...    ││
│  └─────────────────────────────────────────────────────────┘│
│  → 启动时加载到系统 prompt，帮助 Claude 判断相关性            │
└─────────────────────────────────────────────────────────────┘
                              │
                         识别到相关任务
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Level 2: 核心指令层                                         │
│  ┌─────────────────────────────────────────────────────────┐│
│  │ SKILL.md (正文)                                         ││
│  │ - 详细使用说明                                           ││
│  │ - 参数格式                                               ││
│  │ - 最佳实践                                               ││
│  └─────────────────────────────────────────────────────────┘│
│  → Claude 读取完整 SKILL.md 内容                             │
└─────────────────────────────────────────────────────────────┘
                              │
                         特定场景需要
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  Level 3+: 参考文件层                                        │
│  ┌───────────┐  ┌───────────┐  ┌───────────┐               │
│  │reference/ │  │ scripts/  │  │  assets/  │               │
│  │ forms.md  │  │ extract.py│  │ template/ │               │
│  └───────────┘  └───────────┘  └───────────┘               │
│  → 按需访问，保持核心 skill 精简                              │
└─────────────────────────────────────────────────────────────┘
```

**关键洞察**：通过渐进式披露，**skill 能承载的上下文量理论上无上限**。

#### 6.7.3 激活流程

```
1. 初始 Context
   ┌────────────────────────────────────────┐
   │ System Prompt + Skill 元数据 + 用户消息 │
   └────────────────────────────────────────┘
                    │
                    ▼
2. Claude 识别相关 Skill
   → 调用工具读取 SKILL.md
                    │
                    ▼
3. 根据任务需要选择额外文件
   → 读取 reference.md / forms.md 等
                    │
                    ▼
4. 整合知识执行任务
   → 生成结果
```

#### 6.7.4 代码执行优势

Skills 可以打包可执行代码（Python 脚本、Bash 命令），Claude 直接执行而不需要加载到 context：

| 优势 | 说明 |
|------|------|
| **效率** | 用代码执行排序比用 token 生成便宜得多 |
| **可靠性** | 需要确定性结果的操作更稳定 |
| **Context 节省** | 脚本不占用 context window |

**示例**：PDF Skill 包含预写的 Python 脚本提取表单字段，Claude 直接执行，无 context 开销。

#### 6.7.5 开发最佳实践

| 原则 | 做法 |
|------|------|
| **评估优先** | 先用代表性任务测试，识别能力缺口，再针对性构建 skill |
| **结构优化** | 把大 SKILL.md 拆分成多个 reference 文件；互斥/少用的内容分开放 |
| **以 Claude 为中心迭代** | 观察 Claude 实际如何使用 skill；特别关注 name 和 description（决定是否触发） |
| **自省调试** | 当 Claude 偏离预期时，让它反思哪些 context 真正有用 |

#### 6.7.6 安全建议

> "We recommend installing skills only from trusted sources."

安装非官方 skill 前需审计：
- 所有打包的文件
- 代码依赖项
- 外部网络连接（可能导致数据泄露）

#### 6.7.7 当前支持与未来方向

**当前支持平台**：
- Claude.ai
- Claude Code
- Claude Agent SDK
- Claude Developer Platform

**未来规划**：
- 扩展 skill 生命周期（创建、编辑、发现、分享）
- **Agent 自主创建和优化 skill**，把行为模式固化为可复用资源

#### 6.7.8 核心价值

Agent Skills 是一个 **简单格式**，设计目标是 **组织可访问性**：

> 让开发者、团队和终端用户能够通过捕获的专业知识来定制 Agent，而不是为每个用例从头构建新系统。
