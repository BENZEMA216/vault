# LLM-in-Sandbox 论文笔记

> 论文：LLM-in-Sandbox Elicits General Agentic Intelligence
> 链接：https://arxiv.org/pdf/2601.16206
> 阅读日期：2025-01-23

---

## 一、论文概述

### 核心思想

让 LLM 在**代码沙盒（虚拟计算机）**中自由探索，从而在**非代码领域**激发通用智能。

### 一句话总结

**LLM-in-Sandbox = LLM + 虚拟计算机（终端、文件系统、网络）**

---

## 二、沙盒的三大元能力

| 能力 | 说明 | 示例 |
|------|------|------|
| **外部资源访问** | 从互联网获取资源 | `curl`, `pip install`, `requests.get` |
| **文件管理** | 读写和组织数据 | `open()`, `cat`, `grep`, 文件系统操作 |
| **代码执行** | 编写并运行任意程序 | Python 脚本、bash 命令 |

---

## 三、LLM-in-Sandbox 工作流程

```
输入: 任务提示 p, 任务需求 r, 沙盒 S, 最大轮次 T
输出: 最终答案 o

1. 用任务需求 r 配置沙盒 S
2. t ← 0
3. 工具集: {execute_bash, str_replace_editor, submit}
4. while t < T do
5.     模型根据提示 p 和历史记录生成工具调用 a_t
6.     if a_t 是 submit then
7.         break
8.     end if
9.     在沙盒 S 中执行 a_t，获得观察结果 obs_t
10.    将 (a_t, obs_t) 添加到交互历史
11.    t ← t + 1
12. end while
13. 从沙盒提取输出 o（如 /testbed/answer.txt）
14. return o
```

### 三个核心工具

| 工具 | 功能 |
|------|------|
| `execute_bash` | 执行任意终端命令（安装包、管理文件、运行程序） |
| `str_replace_editor` | 创建、查看、编辑文件 |
| `submit` | 表示任务完成 |

## 四、实验结果

### 性能提升（无需额外训练）

在 6 个非代码领域测试，强模型普遍获得提升：

| 领域 | Claude | GPT-5 | DeepSeek |
|------|--------|-------|----------|
| 数学 | +6.6 | +10.1 | +7.9 |
| 物理 | +6.4 | +5.2 | +1.7 |
| 化学 | +1.1 | +0.5 | +1.1 |
| 生物医学 | +1.0 | -6.8 | +2.8 |
| 长文本理解 | +1.3 | +0.5 | +3.0 |
| 指令遵循 | +12.7 | +7.0 | +14.4 |

### 关键发现：上下文放文件系统更好

```
Long-Context 任务准确率：
  - 上下文放 Prompt: 35.6%
  - 上下文放 Sandbox 文件: 48.9%  ← 更好
```

### 强模型 vs 弱模型

| 模型类型 | 外部资源 | 文件管理 | 计算 | 平均轮次 |
|----------|----------|----------|------|----------|
| 强模型 | 6.2% | 21.1% | 12.5% | 12.6 |
| 弱模型 | 0.8% | 2.9% | 2.9% | 23.7 |

弱模型在沙盒里"瞎逛"，轮次多但效率低。

## 五、LLM-in-Sandbox-RL 训练方法

### 核心思想

与普通 LLM-RL（直接在文本上训练）不同，让模型**在沙盒环境中生成轨迹**，学会探索和利用沙盒能力。

### 方法对比

| 方法 | 沙盒交互 | 通用领域数据 | 数据可扩展 | 环境可扩展 |
|------|----------|--------------|------------|------------|
| LLM-RL | ✗ | ✓ | ✓ | N/A |
| SWE-RL | ✓ | ✗ | ✗ | ✗ |
| **LLM-in-Sandbox-RL** | ✓ | ✓ | ✓ | ✓ |

### 关键创新：上下文放入沙盒文件系统

```
多文件上下文：                 单文件上下文 + 干扰项：
/testbed/                      /testbed/
|— documents/                  |— documents/
|   |— introduction.txt ←相关  |   |— report_2021.txt ←相关
|   |— abstract.txt     ←相关  |   |— report_2019.txt ←干扰
|   |— method.txt       ←相关  |   |— report_2020.txt ←干扰
```

**目的**：强迫模型主动探索沙盒环境，而非直接从 prompt 读取答案。

### 奖励设计

| 任务类型 | 奖励函数 |
|----------|----------|
| 多选题 | 正确 → +1，错误 → 0 |
| 自由生成 | ROUGE-L 分数 |
| 数学题 | 正确 → +1，错误 → 0 |

## 七、效率分析

### Token 消耗

长文本任务中，LLM-in-Sandbox 可将 token 消耗降低 **8 倍**（100K → 13K）。

### 吞吐量

环境 token 占 37%-51%，但通过快速 Prefill 处理，整体吞吐量与普通 LLM 模式相当。

### 基础设施开销

- 存储：单一 Docker 镜像 ~1.1GB（vs SWE agents 可达 6TB）
- 内存：每容器 ~50MB 空闲，~200MB 峰值

## 八、个人思考

### 与 Claude Code / OpenCode 的关系

**相同点**：核心思路一致——LLM + 代码沙盒 + 多轮 ReAct

**论文的增量贡献**：
1. 将沙盒范式从**软件工程**扩展到**非代码领域**
2. 提出 **LLM-in-Sandbox-RL** 训练方法
3. 系统性实验验证 + 效率分析
4. 开源 Python 包

### 学术 vs 工程

**结论**：核心范式是工程先行（Claude Code 等产品早就在用），论文更多是学术包装和系统化研究。

## 九、与 Creative CoWork 项目的关联

### 架构对应

Creative CoWork 架构中明确有 **SANDBOX** 模块，与论文思路高度一致：

```
               ┌───────────────────┐
               │    AGENT CORE     │
               └─────────┬─────────┘
     ┌─────────────────┬─┴──────────────────┐
     │                 │                    │
┌────┴────┐     ┌──────┴──────┐       ┌────┴────┐
│  Skills │     │   SANDBOX   │       │  GEN UI │
└─────────┘     └─────────────┘       └─────────┘
```

### 元能力映射

| 论文中的能力 | Creative CoWork 对应 |
|-------------|---------------------|
| 外部资源访问 | 调用各种生成模型 API、下载素材 |
| 文件管理 | 管理项目上下文（素材、中间产物、参考图） |
| 代码执行 | 运行 Skills、ComfyUI workflow、FFmpeg 等 |

### 关键启发

1. **上下文放文件系统**：项目上下文不要全塞进 prompt，让 AGENT 通过文件系统主动检索
2. **多模态生成**：论文 Section 5 展示的图/视频/音频生成正是 Creative CoWork 想做的
3. **AGENT 智能等级**：论文证明现有强 LLM 已具备 L3-L4 能力（上下文利用 + 多步编排）
4. **弱模型训练**：如果用国产/小模型，可参考 LLM-in-Sandbox-RL 训练方法

### 核心痛点呼应

Creative CoWork 的痛点："素材、参考、中间产物、配置散落在 10 个地方"

论文的解法：用沙盒文件系统统一管理，AGENT 通过 `ls`、`grep`、脚本来操作
