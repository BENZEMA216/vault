# VisPainter: From Pixels to Paths

> 论文链接：https://arxiv.org/abs/2510.27452
> 阅读日期：2026-02-02

## 基本信息

- **标题**：From Pixels to Paths: A Multi-Agent Framework for Editable Scientific Illustration
- **作者**：Jianwen Sun, Fanrui Zhang, Yukang Feng, Chuanhao Li, Zizhen Li, Jiaxin Ai, Yifan Chang, Yu Dai, Kaipeng Zhang
- **提交时间**：2025年10月31日
- **领域**：cs.CV

## 一句话总结

基于 MCP 协议的多 Agent 框架，通过操控 Microsoft Visio 生成**可编辑的矢量科学插图**。

---

## 核心问题

现有科学插图生成方法的两大局限：

| 方法 | 问题 |
|------|------|
| 图像生成模型 | 输出栅格图像，缺乏语义结构，无法编辑单独元素 |
| 代码生成（TikZ/SVG） | 提供元素控制，但需要繁琐的「编写-编译-审查」循环 |

**VisPainter 的解法**：操控矢量编辑软件，输出可编辑的矢量文件

## 方法架构

```
┌─────────────────────────────────────────────────────────────┐
│                      用户指令                                │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Manager（管理器）                                           │
│  ├─ 解析用户指令，维护任务状态                                │
│  ├─ 封装请求为 MCP 消息                                      │
│  ├─ 调度工具调用，记录所有交互                                │
│  └─ 支持版本控制、回滚、审计                                  │
└─────────────────────────────────────────────────────────────┘
                              ↓ MCP 协议
┌─────────────────────────────────────────────────────────────┐
│  Designer（设计师）                                          │
│  ├─ 基于 VLM 的「创意引擎」                                   │
│  ├─ 接收 Manager 指令，生成结构化草稿                         │
│  ├─ 指定形状、空间关系                                        │
│  ├─ 接收渲染截图，迭代优化布局                                │
│  └─ 模拟人类「起草 → 精修」流程                               │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  Toolbox（工具箱）                                           │
│  ├─ 执行层：封装 Microsoft Visio 的 GUI/COM 操作             │
│  ├─ 30+ MCP server tools                                    │
│  ├─ 抽象为可调用函数（形状插入、对齐等）                       │
│  └─ 无状态设计，语义决策与实现细节分离                        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  输出：PDF / SVG / VSDX 矢量文件（结构完整，元素可编辑）       │
└─────────────────────────────────────────────────────────────┘
```

### 关键技术选择

| 决策 | 选择 | 理由 |
|------|------|------|
| 协议 | MCP (Model Context Protocol) | 标准化 Agent-Tool 通信 |
| 运行时 | cline runtime | 集成 30+ MCP server tools |
| 矢量编辑器 | Microsoft Visio | 界面简洁，适合 GUI 自动化 |
| 输出格式 | PDF/SVG/VSDX | 保留完整结构，支持元素级编辑 |
| 交互模式 | 原子 GUI 操作 | 无需 DSL 专业知识 |

## VisBench 评估体系

### 7 维度指标

| 层面 | 指标 | 测量方法 |
|------|------|----------|
| **内容** | Precision | 正确渲染的文本元素 / 生成的文本元素 |
| | Recall | 正确渲染的文本元素 / 应有的文本元素 |
| **布局** | Blank-space | 128px 网格覆盖，测量画布利用率 |
| | Alignment | 垂直投影方差分析 |
| **视觉感知** | Design-error | 计数线条、连接器、模块、重叠、溢出错误 |
| | Readability | 考虑遮挡和对比度的文本可读性 |
| **效率** | DQS | Dynamic Quality Score = f(基础分, 交互步数) |

### 数据集

- **规模**：360 个图表
- **来源**：2022-2025 年 STEM 领域论文
- **采样方法**：分层初始化 + 贪婪交换优化，最小化难度方差

## 实验结果

### 模型排名

| 排名 | 模型 | DQS | 特点 |
|------|------|-----|------|
| 1 | Gemini-2.5-Pro | 0.85 | recall 和 readability 最强 |
| 2 | GPT-5 | 0.84 | 接近最高，交互步数较多 |
| 3 | GPT-o3 | - | precision 强，容易漏掉不显眼元素 |
| 4 | Claude-Opus-4 | - | 平衡，但依赖反复自我修正 |
| 5+ | Qwen/Llama | - | 设计保真度不稳定，recall 弱 |

### 消融实验

| 实验 | 结果 | 启示 |
|------|------|------|
| 合并 Manager + Designer | precision -23%, recall -27% | **角色分离是必要的** |
| 步骤粒度 n=1,2,4,8,16 | n≤4 最优，n≥8 性能骤降 | 存在「认知拥堵」上限 |
| Prompt 质量降低 28% | 模型排名稳定 | benchmark 测的是真实能力 |

## 核心洞察

### 1. 可编辑性是本质差异

> 栅格图像 vs 矢量图形不只是格式差异，而是「一次性生成」vs「可持续迭代」的工作流差异

### 2. 角色分离的必要性

> 合并 Manager 和 Designer 导致灾难性下降，说明「规划」和「执行」应该分开

### 3. 认知拥堵现象

> 每轮处理元素数 n≥8 时性能骤降，Agent 的工作记忆有上限

### 4. MCP 协议的价值

> 通过标准化协议封装工具，实现 Agent 与具体实现的解耦

## 与 PaperBanana 对比

| 维度 | VisPainter | PaperBanana |
|------|------------|-------------|
| 输出格式 | 矢量图（可编辑） | 栅格图像 |
| 架构 | Manager-Designer-Toolbox | Agent 流水线 |
| 技术协议 | MCP, 30+ tools | 未披露 |
| 工具 | Microsoft Visio | 图像生成模型 |
| Benchmark | 360 案例, 7 维度 | 292 案例, 4 维度 |
| 评估定义 | 明确量化 | 未明确 |
| 消融实验 | 详细 | 无 |
| 论文质量 | 系统性工作 | 快速验证 |

**结论**：VisPainter 是更扎实的系统性工作，技术细节和实验都更完整。

## 与 Creative CoWork 的关联

### 可借鉴的设计

1. **MCP 协议** — 标准化 Agent-Tool 通信，与 Claude Code 架构一致
2. **角色分离** — Manager（规划）和 Designer（执行）分开
3. **可编辑输出** — 创作工具应输出可持续迭代的格式
4. **步骤粒度控制** — 每轮任务不宜过大（≤4 元素）

### 可参考的评估维度

- 内容层：Precision / Recall
- 布局层：空间利用 / 对齐
- 感知层：错误计数 / 可读性
- 效率层：质量-步数权衡（DQS）

### 局限性

- 依赖 Microsoft Visio（Windows only，商业软件）
- 仅针对科学插图，泛化性待验证
