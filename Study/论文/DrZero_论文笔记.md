# Dr. Zero 论文笔记

> **论文**: Self-Evolving Search Agents without Training Data
> **来源**: Facebook Research
> **仓库**: https://github.com/facebookresearch/drzero

---

## 一、核心思想

**一句话总结**：无需标注数据，让搜索 Agent 通过自我博弈实现能力进化。

传统方法的问题：
- 训练 Agent 需要大量人工标注数据
- 标注成本高、质量难保证
- 数据分布可能与实际场景不匹配

Dr. Zero 的解法：
```
零数据启动 → 自我生成问题 → 自我学习 → 能力涌现
```

---

## 二、双 Agent 架构

| Agent | 角色 | 功能 |
|-------|-----|------|
| **Proposer** | 出题者 | 生成有挑战性但可解的问题 |
| **Solver** | 解题者 | 用搜索工具（如 Wikipedia）回答问题 |

### 2.1 对抗式进化机制

```
┌─────────────┐      生成难题      ┌─────────────┐
│  Proposer   │ ──────────────────▶│   Solver    │
│  (出题者)    │                    │  (解题者)    │
└─────────────┘                    └─────────────┘
       ▲                                  │
       │          反馈信号                 │
       └──────────────────────────────────┘
```

- Solver 变强 → Proposer 被激励出更难的题
- 难题反过来推动 Solver 继续进步
- 形成自动化的课程学习（Curriculum Learning）

### 2.2 关键约束

Proposer 生成的问题必须满足：
1. **可解性**：Solver 有可能答对（不能太难）
2. **挑战性**：略超 Solver 当前能力（不能太简单）
3. **多样性**：覆盖不同类型和难度

---

## 三、技术亮点：HRPO

**Hop-Grouped Relative Policy Optimization**

### 3.1 问题背景

强化学习训练时，每个问题都需要计算 policy gradient，计算量大。

### 3.2 HRPO 的思路

1. **按 Hop 数分组**：将需要相似推理步数的问题聚类
2. **组内相对优化**：在同组内计算相对优势
3. **减少计算开销**：避免跨组比较的无效计算

### 3.3 效果

- 训练效率提升
- 性能与全量计算相当

## 四、工作流程

```
┌────────────────────────────────────────────────────────┐
│                    初始化阶段                           │
│  • Solver: 基础 LLM + 搜索工具                          │
│  • Proposer: 基础 LLM                                  │
│  • 数据: 零标注数据                                     │
└────────────────────────────────────────────────────────┘
                          │
                          ▼
┌────────────────────────────────────────────────────────┐
│                    迭代进化循环                         │
│                                                        │
│  1. Proposer 生成一批问题                               │
│  2. Solver 尝试回答（使用搜索工具）                      │
│  3. 评估答案正确性                                      │
│  4. 用 HRPO 更新 Solver                                │
│  5. 根据 Solver 表现更新 Proposer                       │
│  6. 重复直到收敛                                        │
│                                                        │
└────────────────────────────────────────────────────────┘
                          │
                          ▼
┌────────────────────────────────────────────────────────┐
│                    最终产出                             │
│  • 训练好的 Solver Agent                               │
│  • 具备复杂推理 + 搜索能力                              │
└────────────────────────────────────────────────────────┘
```

## 五、实验结果

论文声称：
- 复杂推理和搜索能力可以仅通过自我进化涌现
- 性能匹配甚至超越有监督基线
- 大幅减少对标注数据的依赖

## 七、适用场景

1. **低资源场景**：没有标注数据但需要训练 Agent
2. **降低成本**：减少人工标注的需求
3. **研究目的**：探索能力涌现的机制
4. **Tool-Use Agent**：训练会使用外部工具的 LLM

---

## 八、局限性分析

| 局限性 | 说明 |
|-------|------|
| **需要 RL 训练** | 不是 Training-free，有训练成本 |
| **Proposer 质量依赖** | 如果 Proposer 生成的问题质量差，进化会卡住 |
| **冷启动问题** | 初始阶段两个 Agent 都很弱，如何启动进化？ |
| **搜索工具依赖** | 需要可靠的外部搜索引擎（如 Wikipedia） |
| **评估困难** | 需要自动判断答案正确性 |

### 8.1 实用性评估：价值有限

**结论：这个工作对工业界几乎没有实用价值。**

原因：

1. **RL 训练门槛高**
   - 需要 GPU 集群、分布式训练框架
   - 需要 RL 调参经验（奖励设计、超参数等）
   - 训练不稳定，容易崩溃

2. **场景太窄**
   - 只针对搜索/问答任务
   - 换个任务类型就要重新设计、重新训练
   - 泛化能力存疑

3. **现成方案更好**
   - 直接用 GPT-4/Claude + RAG，效果可能差不多
   - API 调用成本远低于 RL 训练成本
   - 无需维护训练基础设施

4. **"零数据"是误导**
   - 听起来很美好，实际只是"零标注数据"
   - 仍然需要大量计算资源进行 RL 训练
   - 对于大多数团队，标注数据比 GPU 更便宜

### 8.2 谁可能会用？

- 学术研究者（发论文）
- 大厂 AI Lab（探索新范式）
- **不适合**：创业公司、应用开发者、大多数工程团队

## 九、对 Creative CoWork 的启发

### 9.1 可借鉴的点

- **自我改进循环**：Agent 可以通过自我反馈持续进步
- **对抗式设计**：用"出题者"推动"解题者"进步
- **零数据启动**：减少对用户标注的依赖

### 9.2 需要注意的点

- 自我博弈需要精心设计的奖励信号
- 在创作场景，"正确性"难以自动评估
- 可能更适合有明确答案的任务（搜索/问答）

---

## 十、总结

Dr. Zero 展示了一种有趣的 Agent 训练范式：通过 Proposer-Solver 的自我博弈，在零标注数据的情况下实现能力进化。

**核心贡献**：
- 证明了搜索 Agent 可以"自举"训练
- 提出 HRPO 提升训练效率
- 减少对人工标注的依赖

**与 MemEvolve 的区别**：
- Dr. Zero 进化的是"模型权重"（通过 RL）
- MemEvolve 进化的是"系统代码"（通过 LLM 生成）
- 两者代表了 Agent 自我改进的两条不同路线
