```markdown {wrap}
## Agents 文件夹分析

这个文件夹是 **Anthropic Cookbook** 中关于构建有效智能体的参考实现，基于 Erik Schluntz 和 Barry Zhang 的研究论文《Building Effective Agents》。

### 文件结构

```
/agents/
├── README.md                    # 项目说明文档
├── basic_workflows.ipynb        # 基础工作流程示例 (32KB, 807行)
├── evaluator_optimizer.ipynb    # 评估器-优化器工作流程 (11KB, 306行)
├── orchestrator_workers.ipynb   # 编排器-工作者工作流程 (13KB, 285行)
├── util.py                      # 工具函数库 (1.4KB, 42行)
└── prompts/                     # 提示词模板文件夹
    ├── citations_agent.md       # 引用智能体提示词 (2.8KB)
    ├── research_lead_agent.md   # 研究主管智能体提示词 (23KB)
    └── research_subagent.md     # 研究子智能体提示词 (8.9KB)
```

### 核心组件

#### 1. **工具函数库** (`util.py`)
- `llm_call()`: 统一的 LLM 调用接口，支持 Claude-3-5-Sonnet 模型
- `extract_xml()`: XML 标签内容提取工具，用于解析结构化响应

#### 2. **基础工作流程** (`basic_workflows.ipynb`)
演示了三种基础的多 LLM 工作流程：

- **Prompt Chaining (提示链)**: 将任务分解为顺序子任务，每个步骤基于前一步的结果
- **Parallelization (并行化)**: 将独立子任务分配给多个 LLM 并发处理
- **Routing (路由)**: 根据输入特征动态选择专门的 LLM 路径

#### 3. **高级工作流程**

**评估器-优化器工作流程** (`evaluator_optimizer.ipynb`):
- 一个 LLM 生成响应，另一个提供评估和反馈
- 适用于有明确评估标准和迭代改进价值的场景
- 包含生成、评估、循环优化三个核心函数

**编排器-工作者工作流程** (`orchestrator_workers.ipynb`):
- 中央 LLM 动态分解任务，委托给工作者 LLM，并综合结果
- 适用于复杂任务，其中子任务无法预先预测
- 包含 `FlexibleOrchestrator` 类实现

#### 4. **提示词模板** (`prompts/`)

**研究主管智能体** (`research_lead_agent.md`):
- 23KB 的详细提示词，定义了完整的研究流程
- 支持深度优先、广度优先、直接查询三种查询类型
- 包含子智能体数量指导原则和委托指令

**研究子智能体** (`research_subagent.md`):
- 专门用于执行具体研究任务的智能体
- 支持网络搜索、文档分析等工具

**引用智能体** (`citations_agent.md`):
- 负责处理引用和参考文献的智能体

### 应用场景

这个智能体框架特别适用于：

1. **复杂研究任务**: 需要多角度分析或大量信息收集
2. **内容生成**: 需要多种风格或格式的营销内容
3. **代码开发**: 需要迭代优化和评估的编程任务
4. **客户支持**: 需要智能路由和专业处理的客服场景

### 设计特点

- **模块化设计**: 每个工作流程都是独立的，可以组合使用
- **可扩展性**: 通过提示词模板可以轻松定制智能体行为
- **效率优化**: 支持并行处理和智能任务分配
- **质量保证**: 内置评估和反馈机制确保输出质量

这个文件夹提供了一个完整的智能体开发框架，展示了如何构建高效、可扩展的多 LLM 系统来解决复杂问题。
```

## Prompt 大概内容
```markdown {wrap}
好的，这三个 Markdown 文件分别定义了三种不同角色的智能体（Agent），它们在一个复杂的研究任务中协同工作。下面我为您分别解析它们的内容：

---

### 1. `research_lead_agent.md` (研究主管智能体)

这个文件定义了一个"研究主管"或"项目经理"的角色。它是整个研究任务的**大脑和指挥中心**。

-   **核心职责**:
    1.  **任务理解与规划**: 深入分析用户的请求，制定一个全面的研究策略。
    2.  **任务分解**: 将复杂问题分解为更小的、可执行的子任务。
    3.  **智能体委派**: 将子任务分配给一个或多个"研究子智能体"去执行。
    4.  **结果整合**: 收集所有子智能体的研究成果，进行分析、整合，并最终撰写一份高质量的综合研究报告。

-   **关键工作流程**:
    -   **查询类型判断**: 它首先会判断用户问题的类型是"深度优先"（需要多角度深入研究）、"广度优先"（需要研究多个独立子主题）还是"直接查询"（简单事实查找）。
    -   **动态规划**: 基于查询类型，它会制定不同的研究计划。例如，为"深度优先"问题创建多个从不同视角研究的子智能体。
    -   **并行执行**: 它被要求尽可能地并行创建和运行多个子智能体，以提高效率。
    -   **最终报告撰写**: 它负责最终报告的撰写，但**不负责添加引用**。这个工作会留给专门的引用智能体。

**简单来说，`research_lead_agent` 负责"做什么"和"谁来做"，它自己不进行具体的研究，而是专注于管理整个流程。**

---

### 2. `research_subagent.md` (研究子智能体)

这个文件定义了一个"研究员"或"执行者"的角色。它是研究团队中的**具体执行者**。

-   **核心职责**:
    1.  **接收任务**: 从"研究主管智能体"那里接收一个非常具体、明确的研究任务。
    2.  **执行研究**: 使用各种可用的工具（如网络搜索、网页抓取、内部文档搜索等）来完成分配的任务。
    3.  **信息搜集与批判性评估**: 在搜集信息的过程中，需要评估来源的质量，识别并标记不确定或有争议的信息。
    4.  **提交报告**: 将研究结果整理成一份简洁、信息密集的报告，并提交给"研究主管智能体"。

-   **关键工作流程**:
    -   **工具使用**: 被明确指示要优先使用用户提供的内部工具（如 Google Drive, Slack 等），因为它认为这些工具包含高价值的非公开信息。
    -   **研究循环**: 遵循一个高效的研究循环（观察、判断、决策、行动），并有工具使用次数的预算限制（例如，简单任务少于5次，复杂任务最多15-20次），避免资源浪费。
    -   **源质量评估**: 它被要求对信息来源保持批判性思维，注意区分事实与观点、猜测。

**简单来说，`research_subagent` 是负责"动手做"的智能体，专注于高效、高质量地完成单一、具体的研究任务。**

---

### 3. `citations_agent.md` (引用添加智能体)

这个文件定义了一个"校对"或"格式化"的角色。它负责研究流程的**最后一步**。

-   **核心职责**:
    1.  **接收最终报告**: 从"研究主管智能体"那里接收已经撰写完成的报告文本和所有原始信源。
    2.  **添加引用**: 在报告文本中适当的位置，为关键事实、数据和声明添加正确的引用标记。

-   **关键工作流程**:
    -   **内容不变原则**: 它有非常严格的规定，**绝对不能修改报告的任何文字内容**，包括空格。唯一能做的就是添加引用。
    -   **智能引用**: 它被指导要有选择性地添加引用，只为关键声明、数据或读者可能希望核实的内容添加，而不是为每一句话都添加。
    -   **格式精确**: 输出的文本必须包含在 `<exact_text_with_citation>` 标签中，以确保格式的统一和正确解析。

**简单来说，`citations_agent` 是一个专门负责"收尾工作"的智能体，确保最终报告的学术严谨性和可信度。**

### 总结

这三个文件共同构成了一个分工明确、流程清晰的**多智能体协作系统**：

1.  **`research_lead_agent` (主管)** 运筹帷幄，分解和分配任务。
2.  **`research_subagent` (研究员)** 分头执行，搜集和整理信息。
3.  **`citations_agent` (校对)** 最后加工，为报告添加引用。

这种架构模仿了人类研究团队的工作模式，旨在高效、高质量地完成复杂的研究任务。
```

## Basic Workflow
```markdown {wrap}
好的，这个 Jupyter Notebook (`basic_workflows.ipynb`) 是一个非常经典的多 LLM（大语言模型）工作流的入门教程。它通过简洁的代码和清晰的示例，展示了三种基础但极其强大的方法，来组织多个 LLM 调用以完成单个 LLM 难以解决的复杂任务。

我为您逐一解释：

### 核心目标

这个 Notebook 的核心目标是演示三种基本的多 LLM 工作流模式：
1.  **链式调用 (Prompt-Chaining)**: 像流水线一样，按顺序执行一系列任务。
2.  **并行处理 (Parallelization)**: 像团队分工一样，同时处理多个互不相干的任务。
3.  **路由分发 (Routing)**: 像一个智能调度员，根据任务内容把它分发给最合适的"专家"。

---

### 1. 链式调用 (Prompt-Chaining)

-   **做什么的？**
    将一个复杂任务分解成多个连续的、有先后顺序的子任务。前一步的输出会作为后一步的输入，一步步地对数据进行处理和精炼，直到得到最终结果。

-   **代码实现 (`chain` 函数):**
    这个函数接收一个初始输入 (`input`) 和一个包含多个提示词的列表 (`prompts`)。它会遍历这个列表，用每个提示词去调用 LLM，然后把返回的结果作为下一次调用的新输入。

-   **示例：**
    代码中的示例非常形象：
    -   **输入**: 一段包含各种业绩数据的原始文本报告。
    -   **任务链**:
        1.  **第一步**: 从文本中提取出所有"数值:指标"的键值对。
        2.  **第二步**: 将提取出的数值统一转换成百分比格式。
        3.  **第三步**: 按数值从高到低对所有行进行排序。
        4.  **第四步**: 将排序后的数据格式化成一个精美的 Markdown 表格。
    通过这个"流水线"，一堆杂乱的文本最终变成了一张清晰、有序的表格。

### 2. 并行处理 (Parallelization)

-   **做什么的？**
    当需要对多个不同的输入执行**相同类型**的操作时，这个模式可以把这些任务同时分发给多个 LLM 去处理，大大缩短了总耗时。

-   **代码实现 (`parallel` 函数):**
    这个函数使用 Python 的 `ThreadPoolExecutor` 来创建一个线程池。它接收一个通用的提示词 (`prompt`) 和一个包含多个不同输入的列表 (`inputs`)。它会为每个输入都启动一个独立的 LLM 调用任务，然后等待所有任务完成后，一次性返回所有结果。

-   **示例：**
    代码中的示例是为公司的不同相关方（客户、员工、投资者、供应商）做市场影响分析。
    -   **输入**: 一个包含四个不同利益相关方及其关注点的列表。
    -   **操作**: 对列表中的每一项，都使用同一个指令（"分析市场变化会对这个群体产生什么影响"）去调用 LLM。
    -   **结果**: 无需等待一个分析完再分析下一个，四个分析报告是**同时生成**的，效率极高。

### 3. 路由分发 (Routing)

-   **做什么的？**
    这个模式扮演了一个智能调度中心的角色。它首先会"看一眼"用户的请求，判断这个请求属于哪个领域，然后把它转发给处理该领域问题的"专家 LLM"。

-   **代码实现 (`route` 函数):**
    这是一个两步过程：
    1.  **第一步：分类**。函数首先用一个"选择器提示词 (selector_prompt)"去调用 LLM，让它分析输入内容，并从预设的几个类别（例如：'billing', 'technical', 'account'）中选择一个最合适的。
    2.  **第二步：处理**。在确定了类别后，函数会从一个预设的"专家提示词"字典 (`routes`) 中，找到对应这个类别的专属提示词，然后用这个专家提示词来对原始输入进行第二次 LLM 调用，从而生成更专业、更具针对性的回答。

-   **示例：**
    代码中的示例是一个客户支持系统：
    -   **预设专家**: 定义了四个专家角色：账单支持、技术支持、账户支持、产品支持。每个角色都有自己独特的应答指南。
    -   **处理流程**:
        1.  当收到一个用户工单，如"我无法登录我的账户"。
        2.  `route` 函数先调用 LLM 判断，LLM 会分析后说"这属于账户问题"，并返回 `account`。
        3.  然后函数会使用"账户安全专家"的指南来生成回复，回复内容会非常侧重于安全验证、密码重置等。
        4.  如果工单是"这个月账单为什么多了"，那么它就会被路由给"账单专家"，回复也会侧重于解释费用明细。

### 总结

这个 Notebook 通过非常实际的例子，清晰地展示了如何超越简单的"一问一答"模式，通过**组合和编排**多个 LLM 调用，来构建更强大、更智能、更高效的自动化应用。这三种模式是构建复杂 AI Agent（智能体）的基石。
```

## Orchestrator
```markdown {wrap}
好的，这个 `orchestrator_workers.ipynb` Notebook 展示的是一个比 `basic_workflows` 更高级、更动态的 AI 工作流。它模拟了一个**"项目经理与团队成员"**的协作模式。

### 核心思想：一个"总监"加多个"专员"

想象一个市场营销团队要为一个新产品写宣传文案：

*   **市场总监 (Orchestrator)**：他不会亲自去写每一个字。他的工作是首先理解产品（比如，一个环保水瓶），然后决定这次我们需要几种不同风格的文案来吸引不同的人群。比如，他会说："我们需要一篇给技术宅看的'硬核技术规格版'，一篇给普通大众看的'生活方式故事版'，再来一篇结合两者的'混合版'。"
*   **文案专员 (Workers)**：团队里有几个文案专员。他们会分别领到总监分配下来的具体任务，比如A专员负责写技术版，B专员负责写故事版，C专员负责写混合版。他们各自完成自己的部分。

这个 Notebook 实现的就是这样一个**"总监-专员"**（或称为"编排器-工作者"）的自动化流程。

---

### 工作流程分解

整个过程由 `FlexibleOrchestrator` 这个类来驱动，它包含两个核心步骤：

**第一步：总监进行规划与任务分解 (Orchestrator)**

1.  **接收任务**：`process` 方法接收到一个高级、模糊的任务，例如："为新款环保水瓶写产品描述"。
2.  **调用"总监 LLM"**：它首先使用 `ORCHESTRATOR_PROMPT`（总监的指令）来调用一次 LLM。这个指令的核心是："**请分析这个任务，并把它分解成几个不同的、可执行的子任务**"。
3.  **解析计划**：
    *   "总监 LLM"的返回结果包含两部分：
        *   **`<analysis>`**：它对任务的宏观分析和思考过程（例如，"我认为这个任务需要从技术和情感两个角度切入，因为……"）。
        *   **`<tasks>`**：一个结构化的 XML 列表，里面清晰地定义了每一个子任务的类型 (`<type>`) 和具体描述 (`<description>`)。
    *   代码中的 `parse_tasks` 函数负责把这个 XML 任务列表解析成 Python 程序可以理解的格式。

**第二步：专员们执行具体任务 (Workers)**

1.  **分发任务**：`process` 方法会遍历上一步中由"总监"生成的所有子任务。
2.  **调用"专员 LLM"**：对于每一个子任务，它会使用 `WORKER_PROMPT`（专员的指令）来再次调用 LLM。这个指令的核心是："**这是一个大任务中的一小部分，请根据给定的风格 (`task_type`) 和具体指南 (`task_description`) 来完成这个具体的写作任务。**"
3.  **收集成果**：它会把每个"专员"生成的具体文案 (`<response>`) 收集起来。

**最终产出**

`process` 方法最后会返回一个包含所有工作成果的字典，里面既有"总监"的顶层设计分析，也包含了每个"专员"完成的具体文案。

---

### 和 `basic_workflows` 中的并行模式有何不同？

这是一个非常关键的区别：

*   **简单并行 (Simple Parallelization)**：
    *   子任务是**开发者预先定义好**的。开发者必须在代码里明确写出要并行处理哪几个输入。
    *   **灵活性差**。如果换一个任务，开发者可能需要重写代码来定义新的子任务。

*   **编排器-工作者模式 (Orchestrator-Workers)**：
    *   子任务是**由第一个 LLM（编排器）动态生成**的。开发者只给出一个高层目标。
    *   **灵活性极高**。你给它一个完全不同的任务，比如"为我的播客节目想几个备选标题"，编排器 LLM 就会自动分析并生成新的子任务，比如"生成幽默风格的标题"、"生成悬疑风格的标题"、"生成专业风格的标题"，然后让工作者去执行。

### 总结

这个 **Orchestrator-Workers** 工作流是构建**自适应、有规划能力**的 AI 智能体的核心。它通过一个"规划层"来动态地分解复杂问题，然后再让多个"执行层"去并行处理，这使得 AI 能够应对那些无法预先穷举所有步骤的、更加开放和复杂的现实世界任务。这比简单的链式或并行调用要智能得多。
```
