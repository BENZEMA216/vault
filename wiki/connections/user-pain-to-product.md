# 用户痛点 → 产品特性映射

> 将访谈中发现的真实痛点，映射到 Creative CoWork 的具体产品方案。

## 映射总览

```
用户痛点                    →    Creative CoWork 方案
────────────────────────────    ────────────────────────────
角色一致性差                →    上下文容器 + 素材管理 + 角色知识库
上下文分散                  →    Context Container 统一聚合
工具碎片化                  →    Studio + Skills 系统
抽卡效率低                  →    Agent 多步编排 + 批量生成
单人全链路需求              →    全流程 Agent 工作流
```

## 1. 角色一致性 → 上下文容器 + 素材管理

### 痛点证据

**LSW**（30人团队）：
> 角色一致性是"最大的痛点"。解决方案全靠手工：大量垫图、固定 prompt 模板、多次生成筛选、后期修图。

**三界动画**（100-200人团队）：
> 多角色场景容易"串"（衣服穿错人），3 人以上场景需用 Banana 模型处理。角色换装需创建新资产。AI 无法识别"皇后娘娘"="瑶池公主"——同一角色不同称呼无法关联。

**福州畅读**（出海团队）：
> 底模太少，所有帅哥/老人长一样。比例失调（头大身小）。多人一致性差——衣服乱穿、特征混淆。

### 产品方案

| 痛点细节 | 产品特性 | 概念链接 |
|----------|----------|----------|
| 垫图/参考图管理散乱 | [[concepts/context-container]] 内的素材库：角色三视图、定妆照、参考图统一存储 | 项目级文件管理模块 |
| 角色称呼/别名混乱 | 项目级角色关系图谱：角色定义包含所有称呼映射 | 角色知识库 |
| 多角色场景特征混淆 | Agent 在生成时自动注入当前角色的完整上下文（外观、服装、特征） | [[concepts/agent-loop]] 上下文感知 |
| 换装/换场景资产管理 | 角色资产版本管理：按服装/场景组织多套资产 | 素材版本追踪 |

### 关键设计决策

角色一致性的核心不是"更好的生成模型"，而是**上下文的完整性和可访问性**。当 Agent 在生成每个镜头时都能访问完整的角色定义、历史生成结果、参考素材，一致性自然提升。

## 2. 上下文分散 → Context Container 统一聚合

### 痛点证据

**LSW**：
> "给我一个地方，把素材、参考、生成结果都放在一起，能看到整个项目的全貌。"

**三界动画**：
> 审核在微信群、资产在 NAS、制作在不同平台（即梦、智影）。"希望有统一的流程软件"。

**艺柏AILab**：
> 协作主要在飞书文档 + 分秒帧（视频审阅），5 人小团队分工不明确，大家互相补位。理想形态："像智影的集成度 + Topnote 的协作 + 资产库打通"。

### 产品方案

| 分散场景 | Context Container 解决方式 |
|----------|--------------------------|
| 素材在多个生成工具中 | 统一素材池，支持从即梦/MJ/可灵等导入 |
| 审核在微信群 | 内置审阅流程，结果直接关联到分镜 |
| 文件在 NAS/本地/云端 | 项目级文件系统，支持多来源聚合 |
| 对话历史丢失 | Agent 对话与项目绑定，上下文持久化 |
| 版本管理困难 | 素材和生成结果自动版本追踪 |

### 核心概念链接

- [[concepts/context-container]] — 四大类内容：素材（inputs）、生成结果（outputs）、参考资料（references）、对话历史（conversation history）
- [[concepts/progressive-disclosure]] — Container 内信息按需分层加载，避免上下文过载

## 3. 工具碎片化 → Studio + Skills 系统

### 痛点证据

**福州畅读**：
> "没有一个工具能一站式解决，必须多工具组合。"

**艺柏AILab**（核心工具矩阵跨 10+ 工具）：
> 人设用即梦、场景用 Midjourney、三视图用 Nano、视频用可灵/海螺、换脸用 liblib、配音用嘎嘎或真人、项目管理用文创AI/飞书、视频审阅用分秒帧。

**三界动画**：
> 剧本用扣子智能体、生图用即梦、视频用智影（70%）+即梦（30%）、剪辑用剪映、配音用专业团队或豆包 TTS。

### 产品方案

| 工具碎片 | Skills 系统解决方式 | 概念链接 |
|----------|-------------------|----------|
| 多个生图工具 | 生图 Skill 封装多模型路由（即梦/MJ/Nano），Agent 自动选择最适合的 | [[concepts/skills-system]] |
| 多个生视频工具 | 视频 Skill 封装可灵/智影/海螺等，按场景推荐 | [[concepts/tool-routing]] |
| 剪辑工具独立 | Studio 内置剪辑预览能力，静帧预演不需要切换工具 | Studio 模块 |
| 配音工具独立 | 配音 Skill 集成多 TTS + 真人配音接口 | [[concepts/skills-system]] |

### 关键设计决策

不是重建所有工具，而是通过 [[concepts/skills-system]] 和 MCP 协议**统一接入**。用户在一个界面操作，Agent 在背后路由到最合适的工具。[[concepts/init-mechanism]] 的三层推荐确保用户启动项目时获得最适合的 Skills 组合。

## 4. 抽卡效率低 → Agent 多步编排 + 批量生成

### 痛点证据

**三界动画**：
> 单镜头抽 30-100 次常见。多角色、打戏抽卡量更大。全自动生成浪费算力，错误率高。

**艺柏AILab**：
> 5-10 次不行就改分镜。"不死磕"策略。

**福州畅读**：
> 精品剧自动化程度 < 30%。策略：5-10 次抽不中 → 换模型 → 改分镜。

**关键共识**：分步骤批量 > 一键全自动。用户需要在关键节点确认（Human in the loop 是必须的）。

### 产品方案

| 效率瓶颈 | Agent 编排解决方式 |
|----------|-------------------|
| 单镜头抽卡量大 | Agent 批量生成 + 智能筛选（基于角色一致性/风格匹配自动过滤） |
| 全自动错误率高 | 分步确认模式：生成 → Agent 预筛 → 用户确认 → 下一步 |
| 换模型/改分镜策略 | Agent 感知失败模式，自动建议换模型或调整分镜描述 |
| 不同质量不同策略 | 质量分级编排：低质量全自动、高质量分步确认 |

### 核心概念链接

- [[concepts/agent-loop]] — 创意循环的发散/收敛模式：发散阶段批量生成多种可能性，收敛阶段人工筛选确认
- [[concepts/sub-agent-architecture]] — 主 Agent 编排，生图/筛选/调整分别委托给 Sub-Agent

## 5. 单人全链路 → 全流程 Agent 工作流

### 痛点证据

**三界动画**的关键变化：
> 以前分段制作（分镜、生图、剪辑各有专人），现在一人从头做到尾（分镜→生图→生视频→剪辑）。原因：效率更高，出错率更低，反馈时间减少。

**艺柏AILab**（5人团队）：
> 小团队分工不明确，大家互相补位。导演统筹全局，必要时补位。

**福州畅读**的导演权威：
> "10 个抽卡师，10 种导演风格，上家接上家你都衔接不起来。" 每个导演有自己的情感风格，不能换人。

### 产品方案

| 全链路环节 | Agent 工作流支持 | 视频 Agent 实例 |
|-----------|-----------------|----------------|
| 剧本拆解 | 剧本分析 Agent：角色/道具/场景提取 + 提示词 | [[raw/projects/jimeng-video-agent/[视频Agent-skill]故事短片_构思]] |
| 分镜设计 | 分镜 Agent：景别、机位、画面描述生成 | [[raw/projects/jimeng-video-agent/[视频Agent-skill]故事短片_故事板生成]] |
| 素材生成 | 生图/生视频 Agent：多模型路由 + 批量生成 | [[raw/projects/jimeng-video-agent/[视频Agent-skill]故事短片_参考素材生成（优化版）]] |
| 配乐 | 配乐 Agent：风格匹配 + 搜索词提取 | [[raw/projects/jimeng-video-agent/[视频Agent-skill]故事短片_配乐方案（优化版）]] |
| 剪辑合成 | 剪辑 Agent：物料组装 + 音画同步 | Studio 内置能力 |

### 关键设计决策

全流程不等于全自动。Agent 工作流的价值是让一个人**高效地**走完全链路，而非替代人。每个环节都有人工介入点，符合"分步确认"的用户共识。
